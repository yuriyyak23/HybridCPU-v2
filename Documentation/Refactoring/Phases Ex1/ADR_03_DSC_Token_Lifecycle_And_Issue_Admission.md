# ADR 03 - DSC Token Lifecycle And Issue Admission

Status:
Proposed design gate. Not implementation approval.

Phase:
Phase 03 - DSC Token Lifecycle And Issue Admission.

Decision date:
Not accepted. This document records the required design contract for future
Option B work only.

Scope:
Define the token allocation point, active token ownership model, token
namespace, lifecycle state machine, quota/backpressure behavior, cancellation
rules, and issue/admission contract for future executable lane6
`DmaStreamComputeMicroOp`.

This ADR does not authorize executable DSC. Phase 02 approval remains required
before any CPU/ISE implementation can change lane6 execution behavior.

## Current Contract

The current implemented contract remains unchanged.

- `DmaStreamComputeMicroOp.Execute(ref Processor.CPU_Core core)` remains
  fail-closed.
- `DmaStreamComputeMicroOp` carries descriptor identity, owner/thread/context,
  lane placement, resource mask, normalized read/write footprint evidence, and
  replay evidence. It does not allocate pipeline tokens.
- `DmaStreamComputeDescriptorParser.ExecutionEnabled == false`.
- `DmaStreamComputeRuntime.ExecuteToCommitPending(...)` creates and advances
  tokens inside an explicit helper/model flow. This is not normal pipeline
  issue/admission.
- `DmaStreamComputeToken` currently models states `Admitted`, `Issued`,
  `ReadsComplete`, `ComputeComplete`, `CommitPending`, `Committed`, `Faulted`,
  and `Canceled`.
- `DmaStreamComputeToken.TryAdmit(...)` and the telemetry pressure helpers can
  create model tokens or reject admission, but they are not an active pipeline
  token store.
- Current pressure/admission rejection surfaces include quota, backpressure,
  and token-cap classifications, but they are model/runtime surfaces until
  executable DSC is approved.
- Current token cancellation reasons are narrow model reasons (`Flush`,
  `ReplayDiscard`, `Trap`). They do not yet cover the full future executable
  set such as squash, context switch, reset, guard revocation, or domain
  teardown.
- Current commit uses `Processor.MainMemoryArea` and physical
  `TryReadPhysicalRange(...)` / `TryWritePhysicalRange(...)` through the
  all-or-none helper path.
- No current `DmaStreamComputeTokenStore` owns active executable tokens.
- No current normal pipeline issue/admission path allocates, indexes,
  schedules, completes, cancels, or retires lane6 DSC tokens.
- Compiler/backend production lowering to executable DSC remains forbidden.

## Decision

If Phase 02 later approves Option B, token allocation must happen at
issue/admission.

Issue/admission is the first boundary that can legally combine:

- typed lane6 placement;
- descriptor ABI validation;
- owner/domain guard acceptance;
- normalized read/write footprints;
- resource masks and scheduling legality;
- quota, backpressure, and token-cap checks;
- replay authority;
- future conflict-service admission;
- future backend/address-space admission;
- issuing PC, bundle, slot, virtual thread, context, core, pod, domain, and
  device metadata.

Allocation must not happen at decode. Decode may parse and carry evidence, but
decode is speculative and cannot allocate durable DMA state.

Allocation must not be deferred to retire. Retire is too late to model
backpressure, active footprint conflicts, cancellation, completion, and async
progress.

Memory-stage allocation remains rejected as the default design. It can be
reopened only if a later ADR proves that memory stage is the first
non-speculative boundary with all typed-slot, guard, footprint, quota,
conflict, replay, and backend checks. This ADR recommends issue/admission.

## Token Namespace

Recommended logical namespace:
Globally unique architectural token handles within a running processor
instance and owner generation.

Required properties:

- public token handles must not collide while any old token can still be
  observed, waited, polled, canceled, or retired;
- token IDs must be generated by the token store at issue/admission, not by
  decode and not by descriptor identity alone;
- descriptor identity may seed diagnostics, but it is not a unique active
  token handle;
- implementation may shard storage by pod/core/domain, but lookup must include
  owner metadata so cross-context or cross-domain aliasing fails closed;
- generation/epoch metadata must prevent stale token handles from observing a
  new token after context switch, reset, or token-store reuse;
- token ID uniqueness must be testable under concurrent virtual threads and
  domains.

Minimum token handle fields for future design:

- token ID;
- owner virtual thread ID;
- owner context ID;
- owner core ID;
- owner pod ID;
- owner domain tag;
- device ID;
- generation or epoch.

The exact bit layout is a later implementation detail. The architecture
contract is global logical uniqueness plus owner-scoped authority checks.

## Token Store Contract

Future `DmaStreamComputeTokenStore` is the sole owner of active executable DSC
tokens.

Responsibilities:

- allocate token IDs at issue/admission;
- insert a token only after admission succeeds;
- reject or stall without allocation when quota, backpressure, token-cap, or
  conflict checks fail;
- index active tokens by token ID plus owner metadata;
- expose guarded lookup for poll, wait, cancel, fence, retire, and diagnostics;
- own lifecycle transitions and reject illegal state transitions;
- own active-token capacity and per-domain/per-context quotas;
- support cancellation fan-out by token ID, issuing lane, virtual thread,
  context, domain, core, pod, device, replay epoch, and reset scope;
- prevent a canceled or faulted token from publishing memory;
- prevent a committed token from being canceled;
- retain terminal token evidence until retire/observer cleanup rules release
  it;
- export diagnostics without implying partial successful completion.

The token store must not be a hidden side table populated by decode. It is a
pipeline issue/admission resource.

## Issue/Admission API Contract

Future executable DSC admission must be a single authority boundary. A later
implementation may split helpers internally, but the architectural result must
be equivalent to this contract.

Inputs:

- lane6 `DmaStreamComputeMicroOp` carrier;
- descriptor validation result;
- owner/domain guard decision;
- issue/admission placement facts;
- issuing PC and bundle identity;
- slot/lane identity;
- virtual thread, context, core, pod, domain tag, and device ID;
- normalized read/write footprints and footprint hash;
- resource masks and scheduler legality evidence;
- replay phase/epoch facts;
- pressure snapshot and token quota policy;
- future conflict-service admission result;
- future backend/addressing admission result.

Outputs:

- accepted with allocated token handle and token-store entry;
- deterministic stall/retry with no token allocation;
- telemetry-only reject with no token allocation;
- architectural fault with no durable token unless the precise-fault ADR
  explicitly requires a retained fault token.

The accepted path must be atomic with respect to token-store capacity. If the
store cannot insert the token, admission fails without partially allocated
state.

## State Machine

Future executable DSC must preserve the current model state names unless a
later ADR explicitly renames them.

Legal lifecycle:

| State | Meaning | Allowed next states |
|---|---|---|
| `Admitted` | Token allocated and stored after issue/admission acceptance. No work has been dispatched. | `Issued`, `Faulted`, `Canceled` |
| `Issued` | Scheduler/engine owns active work. Reads may be in flight. | `ReadsComplete`, `Faulted`, `Canceled` |
| `ReadsComplete` | Source reads are complete. Destination writes are still staged only. | `ComputeComplete`, `Faulted`, `Canceled` |
| `ComputeComplete` | Compute is complete and staged write coverage is being checked. | `CommitPending`, `Faulted`, `Canceled` |
| `CommitPending` | All-or-none staged result is ready for retire/commit observation. No architectural memory publication yet. | `Committed`, `Faulted`, `Canceled` |
| `Committed` | Architectural commit completed at the approved retire/commit boundary. | terminal |
| `Faulted` | Token has a fault record. No memory publication is allowed. | terminal until retire/cleanup |
| `Canceled` | Token was canceled before commit. Staged writes are discarded. | terminal until cleanup |

Rules:

- `Admitted`, `Issued`, `ReadsComplete`, `ComputeComplete`, and
  `CommitPending` are not memory-publication states.
- Only the approved retire/commit path can move `CommitPending` to
  `Committed`.
- Fault and cancellation clear staged writes or mark them non-publishable.
- `Committed` is final and cannot be canceled.
- `Faulted` cannot commit.
- `Canceled` cannot commit.
- Any late engine completion after `Canceled` or `Faulted` must be ignored or
  converted into diagnostics, never memory publication.
- Progress counters and staged byte counts are diagnostic only and do not
  create partial success.

## Cancellation Contract

Future executable DSC must define cancellation before implementation. Minimum
required cancellation causes:

- explicit cancel by token ID;
- replay discard;
- speculative squash;
- trap or precise exception redirect;
- context switch;
- reset;
- guard revocation;
- domain teardown;
- device/backend reset;
- token-store eviction during fatal teardown;
- fence/order policy violation before commit.

Cancellation rules:

- cancellation before `Committed` must prevent memory mutation;
- cancellation after commit begins must be serialized with the commit/retire
  owner and either fail closed before publication or complete the already
  committed architectural operation;
- cancellation must remove the token from active scheduling and clear staged
  writes;
- cancellation must publish diagnostics and cleanup evidence according to the
  retire/observer contract;
- cancellation due to replay/squash/trap/context switch must not leak work into
  a later owner generation;
- guard revocation or domain teardown must cancel all matching active tokens
  before any later wait/poll can observe stale success.

The current `DmaStreamComputeTokenCancelReason` enum is insufficient for the
full future set and must be extended or mapped by a future implementation
phase.

## Quota, Backpressure, And Admission Failure

Quota/backpressure is an admission concern, not a decode concern.

Future executable mode must define:

- global active token cap;
- per-core and per-pod caps;
- per-domain and per-context caps;
- per-device cap if backend/device scheduling is enabled;
- deterministic ordering when multiple lanes compete for a token;
- whether backpressure stalls, retries, or returns a non-architectural reject;
- which failures are telemetry-only and which are architectural faults;
- how outstanding-token counts are updated atomically with token insertion and
  cleanup.

Rules:

- quota/backpressure/token-cap reject must not allocate a token;
- pressure rejects must be deterministic for conformance;
- descriptor, owner/domain, permission, translation, alignment, and ABI faults
  are not pressure rejects and must be routed to the precise-fault contract;
- token-cap diagnostics must not imply successful partial completion or hidden
  scheduler progress.

## Retire And Completion Interaction

Phase 04 owns precise retire fault publication. This ADR fixes the token data
that Phase 04 needs.

Every executable token must retain:

- issuing PC;
- bundle identity or serial;
- slot/lane index;
- virtual thread ID;
- owner context ID;
- owner core and pod ID;
- owner domain tag;
- device ID;
- descriptor identity hash;
- normalized footprint hash;
- issue cycle;
- completion cycle if completed;
- cancellation reason if canceled;
- fault record if faulted;
- replay/epoch generation.

`CommitPending` means "ready for the approved commit/retire owner to observe",
not "memory has been written". Memory publication is a later retire/commit
action and remains all-or-none unless a separate future ADR approves partial
success.

## Exact Non-Goals

This ADR does not:

- implement `DmaStreamComputeTokenStore`;
- change `DmaStreamComputeMicroOp.Execute(...)`;
- change `DmaStreamComputeDescriptorParser.ExecutionEnabled`;
- turn helper-created tokens into pipeline tokens;
- make `DmaStreamComputeRuntime.ExecuteToCommitPending(...)` canonical ISA
  execution;
- allocate tokens at decode;
- publish memory from admitted, issued, read-complete, compute-complete, or
  commit-pending states;
- define final precise fault priority; Phase 04 owns that;
- define final memory ordering, waits, polls, or fences; Phase 05 owns that;
- define physical/IOMMU backend selection; Phase 06 owns that;
- define cache flush/invalidate protocol; Phase 09 owns that;
- approve compiler/backend production lowering.

## Implementation Phases Enabled Only After Approval

Approval of this ADR would enable later implementation planning for:

1. `DmaStreamComputeTokenStore` as an active-token owner.
2. Issue/admission token allocation API.
3. Token handle generation and guarded lookup.
4. Quota/backpressure/capacity admission policy.
5. Extended token metadata for retire, replay, cancellation, and diagnostics.
6. Cancellation fan-out hooks from replay, squash, trap, context switch,
   reset, guard revocation, and domain teardown.
7. Scheduler/engine integration that advances existing token states.
8. Retire observation handoff to Phase 04.

These workstreams remain blocked from code implementation until Phase 02 and
this Phase 03 design are both approved.

## Tests Required Before Implementation

Before any executable token claim, tests must cover:

- no token allocation on decode-only bundles;
- no token allocation for squashed decode/speculative bundles;
- exactly one token allocation per accepted issue/admission event;
- token IDs unique in the approved namespace across virtual threads, contexts,
  domains, cores, and pods;
- descriptor identity alone cannot alias active token handles;
- owner/domain mismatch rejects admission without token allocation;
- quota/backpressure/token-cap rejection is deterministic and leaves no active
  token behind;
- accepted token carries issuing PC, bundle, slot/lane, VT, context, core, pod,
  domain, device, descriptor identity, and footprint metadata;
- cancel before commit prevents memory mutation;
- faulted token cannot commit;
- canceled token cannot commit;
- committed token cannot be canceled;
- late completion after cancellation cannot publish memory;
- context switch handling follows the approved preserve-or-cancel rule;
- reset/domain teardown cancels matching active tokens and invalidates stale
  handles;
- progress diagnostics do not report partial success;
- helper-created model tokens are not counted as pipeline-issued tokens.

## Documentation Migration Rule

Until implementation and tests land, documentation must say:

- current tokens are model/runtime surfaces;
- current helper-created tokens are not pipeline tokens;
- future executable tokens allocate at issue/admission only after approval;
- future token store is required before executable DSC;
- progress diagnostics are not partial successful completion.

Current Implemented Contract must not say:

- lane6 DSC allocates tokens during normal pipeline issue;
- decode allocates durable DSC tokens;
- `DmaStreamComputeRuntime.ExecuteToCommitPending(...)` is pipeline issue;
- helper token telemetry is an active executable token store;
- token `CommitPending` means memory is already visible;
- compiler/backend may assume executable DSC tokens exist.

## Code Evidence

Current token model:

- `HybridCPU_ISE\Core\Execution\DmaStreamCompute\DmaStreamComputeToken.cs`
  - `DmaStreamComputeTokenState` defines `Admitted`, `Issued`,
    `ReadsComplete`, `ComputeComplete`, `CommitPending`, `Committed`,
    `Faulted`, and `Canceled`.
  - `DmaStreamComputeToken.TryAdmit(...)` creates/admission-gates model tokens
    from validation results.
  - `MarkIssued()`, `MarkReadsComplete()`, `StageDestinationWrite(...)`,
    `MarkComputeComplete()`, `Cancel(...)`, `Commit(...)`, and fault
    publication APIs model lifecycle transitions.
  - `Cancel(...)` rejects cancel after `Committed`, clears staged writes, and
    marks the token `Canceled`.
  - `Commit(...)` rejects canceled/faulted tokens and requires
    `CommitPending`.
  - `TryCommitAllOrNone(...)` uses physical main memory and rollback logic.

Current helper/model token creation:

- `HybridCPU_ISE\Core\Execution\DmaStreamCompute\DmaStreamComputeRuntime.cs`
  - `ExecuteToCommitPending(...)` is explicitly documented as helper/model
    code not wired into `DmaStreamComputeMicroOp.Execute`.
  - The helper creates a token, calls `MarkIssued()`, reads operands, computes,
    stages writes, and reaches `CommitPending` or a fault result.

Current carrier evidence:

- `HybridCPU_ISE\Core\Pipeline\MicroOps\DmaStreamComputeMicroOp.cs`
  - Carries descriptor identity and normalized footprint hash.
  - Carries owner virtual thread and owner context information.
  - Publishes normalized read/write ranges into carrier metadata.
  - Builds resource masks for DMA, StreamEngine, load/store, and memory-domain
    buckets.
  - `Execute(...)` remains fail-closed and does not allocate a token.

Current pressure/admission evidence:

- `HybridCPU_ISE\Core\Execution\DmaStreamCompute\DmaStreamComputeValidationResult.cs`
  - Defines `QuotaAdmissionReject`, `BackpressureAdmissionReject`, and
    `TokenCapAdmissionReject`.
- `HybridCPU_ISE\Core\Execution\DmaStreamCompute\DmaStreamComputeTelemetry.cs`
  - Defines pressure policy and snapshot fields including
    `OutstandingTokenCap` and `OutstandingTokens`.
  - Tracks accepted, active, staged, committed, canceled, and faulted token IDs
    for telemetry.
  - Pressure-aware `TryAdmit(...)` can reject before token creation.
  - This is evidence for future design, not current pipeline token allocation.

Current pipeline/replay/retire surfaces:

- `HybridCPU_ISE\Core\Pipeline\Core\CPU_Core.Pipeline.cs`
  - `FlushPipeline(...)` cancels in-flight explicit memory requests and
    invalidates replay context. Future DSC tokens need cancellation fan-out
    from this class of events.
- `HybridCPU_ISE\Core\Pipeline\Certificates\ReplayPhaseSubstrate.cs`
  - Provides replay invalidation vocabulary including DSC-specific
    descriptor/carrier mismatch reasons.
- `HybridCPU_ISE\Core\Pipeline\Core\CPU_Core.PipelineExecution.Memory.cs`
  - `ApplyRetiredDmaStreamComputeTokenCommit(...)` exists as an explicit
    retired token commit helper. It does not prove normal pipeline token issue
    or async completion observation.
- `HybridCPU_ISE\Core\Pipeline\Scheduling\MicroOpScheduler.Admission.cs`
  - Existing scheduler admission and replay-aware placement surfaces are
    candidate integration points. They do not allocate DSC tokens today.

## Strict Prohibitions

Do not claim any of the following until code and tests prove them:

- lane6 DSC allocates tokens in normal pipeline issue;
- decode allocates durable DSC tokens;
- helper-created tokens are executable pipeline tokens;
- a token store exists for executable DSC;
- async token scheduling is implemented;
- token `CommitPending` publishes memory;
- canceled or faulted tokens can commit;
- partial completion is a successful architectural mode;
- compiler/backend may production-lower to executable DSC.

## Exit Criteria

This Phase 03 design can be marked approved only when the approval record
includes:

- final token allocation point;
- final token namespace and stale-handle prevention rule;
- token store ownership and lookup contract;
- issue/admission API inputs and outputs;
- state transition table;
- metadata required by retire/fault publication;
- quota/backpressure/capacity policy;
- cancellation causes and fan-out contract;
- helper/model token compatibility rule;
- required tests and documentation migration rule.

Until then, this file is a design gate only.
