# План рефакторинга сборки для второго (копии) проекта: причина 22 000 ошибок и пошаговое исправление

## 1) Краткий диагноз

Массовые ошибки (десятки тысяч) были **каскадными**, а не первичными по коду.
Первопричина — поломка цепочки `ProjectReference` для `HybridCPU_ISE` в режиме test-support, после чего компилятор не видел нужную сборку и пространства имён.

Типичный первичный симптом:

- `CS0006`: не найден файл метаданных `HybridCPU_ISE.dll` (или reference-assembly)

После этого идут производные ошибки:

- `CS0234` (пространство имён не существует)
- `CS0246` (тип не найден)
- и далее лавинообразно по всем тестам/проектам

Именно поэтому количество ошибок выглядит как `22 000+`: это не 22 000 отдельных дефектов, а 1–2 инфраструктурных сбоя сборки.

---

## 2) Что было причиной в конфигурации

### 2.1 Жёстко зашитый режим `Debug`

В `AdditionalProperties` для `ProjectReference` использовалось фиксированное:

- `Configuration=Debug`
- `OutputPath=bin\Debug\testsupport\`

Это ломает согласованность, если решение/проект строится в другом контексте (или частично с другими свойствами).

### 2.2 Конфликт промежуточных артефактов `obj`

Для ссылки на `HybridCPU_ISE` переопределялись:

- `BaseIntermediateOutputPath=obj\testsupport\`
- `MSBuildProjectExtensionsPath=obj\testsupport\`

Из-за этого мог не находиться `project.assets.json` в ожидаемом месте, и возникал `NETSDK1004`.

### 2.3 Побочный эффект: пропажа reference assembly

Когда обычная и test-support сборки начинают пересекаться по промежуточным файлам или выполняются не в ожидаемом порядке, появляется ошибка вида:

- не найден `obj\Debug\net10.0\ref\HybridCPU_ISE.dll`

Это снова вызывает каскад `CS0234/CS0246`.

### 2.4 Неглавная, но частая помеха

Отдельно может всплывать блокировка файла (`CS2012`) антивирусом/другим процессом. Это не логическая ошибка проекта, а внешняя блокировка файлов сборки.

---

## 3) Целевой принцип для копии проекта

Для второго (копии) проекта нужно сохранить три правила:

1. **Без абсолютных путей** в конфигурации сборки.
2. Для `ProjectReference` передавать test-support свойства через относительные пути и `$(Configuration)`.
3. Разводить test-support артефакты в `obj\testsupport\`, но restore-метаданные оставлять в стандартном `obj\`.

---

## 4) Эталонный шаблон свойств для копии проекта

В проектах, где есть `ProjectReference` на `HybridCPU_ISE`, использовать такой паттерн:

- Базовое значение:
  - `DefineTestSupport=$(DefineTestSupport)`
- Для test-support (`Condition="'$(DefineTestSupport)' == 'true'"`):
  - `DefineTestSupport=$(DefineTestSupport)`
  - `BaseIntermediateOutputPath=obj\testsupport\`
  - `MSBuildProjectExtensionsPath=obj\`
  - `OutputPath=bin\$(Configuration)\testsupport\`

Ключевой момент:

- `MSBuildProjectExtensionsPath=obj\` (а не `obj\testsupport\`), чтобы `project.assets.json` создавался там, где его ожидает SDK.

## 5) План рефакторинга для второго (копии) проекта

### Шаг 1. Зафиксировать точку отказа

1. Выполнить сборку копии проекта.
2. Выделить **первую** ошибку в логе (обычно `CS0006` или `NETSDK1004`).
3. Не анализировать тысячи последующих ошибок до устранения первой.

Ожидаемый результат: определён реальный первичный сбой, а не каскад.

### Шаг 2. Найти все ссылки на `HybridCPU_ISE` с `AdditionalProperties`

Проверить все `.csproj` в копии:

- тестовые проекты
- консольные утилиты тестовой сборки
- промежуточные мосты/интерфейсные проекты
- проекты компилятора

Ожидаемый результат: полный список мест, где задаются свойства для ссылки на `HybridCPU_ISE`.

### Шаг 3. Убрать жёсткие значения конфигурации

Везде заменить фиксированный `Debug` на `$(Configuration)`.

Ожидаемый результат: сборка не зависит от вручную зашитого режима.

### Шаг 4. Привести `OutputPath` к относительному виду

Использовать `bin\$(Configuration)\testsupport\`.

Ожидаемый результат: предсказуемый выходной путь без абсолютных привязок.

### Шаг 5. Исправить путь SDK-метаданных

Если включён `BaseIntermediateOutputPath=obj\testsupport\`, то одновременно выставить:

- `MSBuildProjectExtensionsPath=obj\`

Ожидаемый результат: исчезает `NETSDK1004` по `project.assets.json`.

### Шаг 6. Очистить stale-артефакты

Очистить `bin/obj` у затронутых проектов (в копии), затем сделать `Restore` и `Build`.

Ожидаемый результат: сборка идёт на чистом состоянии без артефактов старой схемы.

### Шаг 7. Проверить изолированно и затем целиком

1. Сначала собрать проблемный проект изолированно.
2. Затем собрать всё решение.

Ожидаемый результат: нет `CS0006/NETSDK1004`; каскад `CS0234` исчезает.

### Шаг 8. Обработать внешние блокировки файлов

Если остаётся `CS2012` (файл занят), проверить фоновые процессы/антивирус, повторить сборку после освобождения файла.

Ожидаемый результат: устранены инфраструктурные блокировки I/O.

## 6) Чек-лист «готово» для копии проекта

Считать исправление завершённым, если выполнены все пункты:

- [ ] Нет абсолютных путей в правках `.csproj`
- [ ] Нет фиксированного `Configuration=Debug` в `AdditionalProperties`
- [ ] `OutputPath` использует `bin\$(Configuration)\testsupport\`
- [ ] При `BaseIntermediateOutputPath=obj\testsupport\` установлен `MSBuildProjectExtensionsPath=obj\`
- [ ] После очистки и восстановления не возникает `NETSDK1004`
- [ ] Нет `CS0006` по `HybridCPU_ISE.dll`/`ref`-сборке
- [ ] Каскад `CS0234/CS0246` исчез

---

## 7) Почему это даёт эффект именно на «22 000 ошибок»

Потому что у вас нарушалась основа графа сборки: отсутствовала или некорректно построенная зависимость `HybridCPU_ISE`.
Как только восстанавливается корректная поставка этой зависимости (метаданные + outputs), большая часть «ошибок кода» автоматически исчезает как вторичный шум.

---

## 8) Рекомендация по стандартизации на будущее

Чтобы проблема не вернулась в следующих копиях/ветках:

1. Вынести общие свойства test-support в единый `.props`/`.targets` (относительные пути).
2. Для всех потребителей `HybridCPU_ISE` использовать один и тот же шаблон `AdditionalProperties`.
3. Проверять first-failure policy в CI: падать по первой инфраструктурной ошибке (`CS0006/NETSDK1004`) до анализа каскада.

Это снизит риск повторного «взрыва» количества ошибок при миграциях и копировании проекта.
