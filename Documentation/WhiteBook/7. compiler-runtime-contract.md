# Compiler Runtime Contract

## Goal of this chapter

This chapter documents the current compiler/runtime contract: versioning, metadata transport, typed-slot facts, agreement checks, and the boundary between compiler intent and runtime authority.

## Versioned contract

The authoritative contract sentinel is `CompilerContract.Version`, currently `6`.

`CompilerContract.ThrowIfVersionMismatch(...)` is fail-closed. A stale producer version is rejected rather than accepted with degraded semantics.
The legacy static-runtime compiler bootstrap now publishes that version once at `Processor.InitializeLegacyCompatRuntime` before compat bridge recording is allowed to append instructions.

The current version history in code explicitly records that:

- legacy scheduling policy in `word3[50]` is retired from the correctness path;
- `VirtualThreadId` in `word3[49:48]` remains a transport hint only;
- all scheduling policy is expected to live in sideband metadata rather than in architectural slot bits.

## Contract layers

The current contract is layered rather than monolithic:

1. Architectural payload contract: raw VLIW slot/bundle bits
2. Sideband metadata contract: slot and bundle annotations
3. Structural agreement contract: typed-slot facts and runtime validation
4. Dynamic authority contract: runtime legality and placement remain authoritative

## Sideband metadata transport

The main current sideband transport types are:

- `InstructionSlotMetadata`
- `IrTypedSlotAdmissionDescriptor`
- `VliwBundleAnnotations`
- `BundleMetadata`

`InstructionSlotMetadata` pairs a transport `VtId` with core `SlotMetadata`. `HybridCpuThreadCompilerContext` now publishes a producer-side admission subset there as well: stealability, opcode-derived placement (`RequiredSlotClass`, runtime pinning kind, pinned lane when applicable), and the current thread `DomainTag`. This remains a transport/validation surface only; register and memory facts are still derived later from IR/decode truth. `IrSlotMetadata` then carries the canonical `IrTypedSlotAdmissionDescriptor` once IR construction derives it from opcode semantics: resource class, required runtime `SlotClass`, binding kind, and legal slot mask. `VliwBundleAnnotations` carries an array of per-slot metadata and derives bundle metadata only when explicit non-default slot metadata exists.

The legacy `StealabilityHint` survives only as advisory metadata for bounded compiler heuristics and diagnostics. It is not the central explanation surface for compiler/runtime agreement, and it does not affect typed-slot structural admissibility or lane binding. Thread-level `DomainTag` changes now invalidate cached canonical compile artifacts so the mainline compiled program cannot silently retain stale producer-side domain facts.

## Typed-slot facts

The compiler/runtime shared vocabulary is `TypedSlotBundleFacts`. Current runtime validation checks:

- per-slot class matches live placement
- per-class counts match live bundle contents
- pinned-lane uniqueness
- class counts stay within topology capacity
- pinned and flexible totals match live operation totals

This agreement model exists today in runtime code even though the runtime can still execute canonical bundles without mandatory compiler-emitted slot metadata.
The dedicated artifact `22. typed-slot-contract-staging.md` freezes the staged status of this contract: current mainline mode is `ValidationOnly`, while `WarnOnMissing` and `RequiredForAdmission` are reserved future vocabulary rather than active policy.
At the bridge layer, `CompilerContract.CurrentTypedSlotPolicy` is `CompatibilityValidation`: missing facts remain compatible, structural agreement failures are quarantine-logged instead of rejected, and `StrictVerification` is available only as an explicit stricter verification mode. The required-for-admission policy remains a future seam rather than active runtime behavior.
The canonical compiled-program artifact now carries `HybridCpuCompiledProgram.AdmissibilityAgreement`, and canonical compilation emits an `AgreementSummary` progress signal with bundle count, structural agreement failures, typed-slot validity, and safety-mask conflict counts. This makes agreement failures visible to build and diagnostic consumers without turning compiler metadata into runtime legality authority.
Parallel decomposition now exposes the same aggregate truth surface through `ParallelCompilationResult.OverallAgreement`, and its `AllTypedSlotFactsValid` / `AllProgramsAdmissible` aggregates include the coordinator program rather than silently reporting worker-only success.
The dedicated artifact `21. reject-taxonomy-matrix.md` closes the next question that follows from this contract: which compiler-side inadmissibility buckets do and do not have runtime reject twins. The canonical code seam for that closure is `AdmissibilityRuntimeVocabulary`, which publishes only proven direct twins and keeps compatibility-adjacent or compiler-only families explicit.

## Runtime guarantees

The current runtime guarantees that:

- class admission happens before lane binding on the typed-slot path;
- lane selection is deterministic;
- legality decisions remain runtime-owned;
- pipeline correctness for canonical opcodes does not depend on non-default slot metadata;
- typed-slot disagreement is visible as bridge agreement evidence before any stricter rejection policy is selected;
- version mismatch is fail-closed.

That is an intentionally asymmetric contract. The compiler may supply structure, but the runtime does not delegate correctness to stale or missing metadata.
`AdmissibilityClassification.StructurallyAdmissible` therefore means "compiler preflight passed", not "the runtime is forbidden from issuing later dynamic rejects."

## Instruction-refactor compiler boundary

The completed instruction-refactor phases sharpen the compiler/runtime boundary:

- the live compiler emits the tracked scalar repair tail through `ZEXT.W = 321`;
- `PlatformAsmFacade.VSetVli` is a scoped positive compiler contour;
- the existing coordinator barrier emission is the canonical zero-payload `FENCE` form;
- Lane6 and Lane7 compiler paths are descriptor-carrier transport only;
- control-flow facade wrappers must remain fail-closed until relocation-aware lowering is implemented;
- atomic lowering, masked/general `FENCE` lowering, `FENCE_I` helper lowering,
  broad vector production lowering, cache/TLB/coherency lowering, and broad
  lane6/lane7 lowering beyond current gated contours remain separately gated.
- MatrixTile positive compiler emission already exists in this checkout, but it
  remains downstream of runtime-owned opcode, resource, placement, legality,
  execute/capture, retire, and replay authority.

Compiler emission is therefore a scope signal for live-emitted forms, not execution authority. The runtime-owned decode/materialization/execution/retire path remains the correctness boundary.

## Specialized compiler/runtime boundaries

The specialized WhiteBooks add explicit non-inversion rules:

- VMX vocabulary may be emitted or decoded as frozen compatibility ABI, projection, or denied/fail-closed vocabulary only. VMX compiler surfaces must not reintroduce `VmxExecutionUnit`, `VmcsManager`, `IVmcsManager`, active VMCS pointer state, VMCS field stores, or VMX-owned SecureCompute grants.
- Lane6 descriptor payload travels through typed sideband. Raw scalar fields, reserved bits, and raw VT hints are not ABI authority. DSC2 parser/model evidence and helper/token observations do not authorize queue lifecycle, async overlap, coherent DMA/cache, or broad production lowering.
- MatrixTile handoff metadata carries `MatrixTileMemory` or
  `MatrixTileCompute`, required slot class, and lane mask. Compiler emission
  cannot reinterpret MTILE memory as ordinary LSU or select DSC/VMX/backend
  fallback.
- L7-SDC `ACCEL_*` carriers are lane7 system-device command carriers. Descriptor, capability, token, queue, backend, fence, telemetry, and commit-helper surfaces do not by themselves grant architectural publication or compiler lowering authority.
- SecureCompute compatibility advertisement is open. VMX, VMCS, VMREAD, VMWRITE, and `VmxCaps` cannot activate, grant, checkpoint, migrate, materialize, or own SecureCompute authority.

## Test evidence

The strongest explicit contract test surface is `Phase12CompilerContractHandshakeTests.cs`, which verifies that:

- the compiled program publishes the current contract version;
- the compiled program publishes and preserves its agreement summary;
- canonical compile progress reports agreement failures as a first-class build signal;
- the legacy compiler bootstrap binds the bridge handshake before compat recording starts;
- duplicate bridge handshake declarations are rejected once the ingress path is already bound;
- stale producer versions are rejected by runtime/compiler bridge surfaces.

## Current limitations

- The current runtime still treats missing or default slot metadata as valid for canonical execution.
- The typed-slot staging contour is now published in `22. typed-slot-contract-staging.md`, but the broader contract still spans runtime types and source comments.
- Compiler-side fact emission is not yet the sole mainline operating mode.

## Open questions / future work

- When typed-slot facts should become default compiler output.
- Whether bundle-level agreement should be externalized into a more stable artifact format.
- Whether the contract should be split into "architectural carrier" and "runtime optimization metadata" documents.
