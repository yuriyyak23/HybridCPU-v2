# Execution Model

## Goal of this chapter

This chapter describes the current runtime execution contour from fetch to retire, including where densification and replay interact with canonical decode, and where deterministic boundaries are enforced.

The companion artifact `Documentation/operational-semantics.md` externalizes this contour as an explicit machine-state tuple and cycle-step relation.

## Pipeline shell

`Processor.CPU_Core` currently implements a five-stage pipeline shell:

1. Fetch (`IF`)
2. Decode (`ID`)
3. Execute (`EX`)
4. Memory (`MEM`)
5. WriteBack (`WB`)

`ExecutePipelineCycle()` advances these stages in reverse order each cycle (`WB -> MEM -> EX -> ID -> IF`) so each stage sees prior-cycle state.

## Fetch contour

Fetch first checks replay opportunity through the loop buffer:

- if loop-buffer replay hits, the core reuses cached decoded micro-ops and does not refetch a raw 256-byte bundle;
- if replay misses, the core fetches 256 bytes from memory/cache and carries optional bundle annotations forward.

This creates a clear split between raw bundle ingress and replay-based decoded-bundle reuse.

## Decode contour

Decode is full-bundle aware, even though foreground issue later advances slot by slot. The decode stage:

- decodes the whole bundle into canonical descriptors;
- may load the loop buffer for multi-iteration replay;
- may apply SMT densification when enabled;
- preserves canonical decode state separately from derived issue-plan state;
- skips empty/NOP or non-issue foreground slots through explicit foreground logic.

The runtime keeps `decodedBundleRuntimeState` and `decodedBundleDerivedIssuePlanState` separate so that densification does not overwrite canonical front-end truth.

## Issue and materialization boundaries

The execution path contains a strong boundary between:

- legality/admission decision;
- lane materialization;
- stage-latch materialization.

`MicroOpScheduler` decides whether a candidate can enter the bundle and which lane it gets. Later, `CPU_Core.PipelineExecution.Materialization.cs` turns issue-packet lanes into concrete EX, MEM, and WB stage lane state.

This is important architecturally because "bundle contains a slot" and "slot becomes a live execution lane" are not the same boundary.

## Execution-surface contract

The runtime explicitly guards configured execution surfaces. `CPU_Core.PipelineExecution.SurfaceContract.cs` rejects:

- unsupported mainline stream-control surfaces;
- VMX slots that reach execution materialization without a wired VMX execution plane.

So the execution model uses explicit reject boundaries at route-to-surface
checks. This should not be read as a blanket fail-closed claim beyond the
ingress/version handshake and the named execution-surface guard seams.

## Deterministic principles

The current execution model aims for deterministic behavior through explicit mechanisms:

- canonical decode is table-driven and fail-closed;
- lane selection is deterministic and replay-aware;
- serializing boundaries flush or invalidate replay-sensitive state;
- class templates and legality templates are invalidated when replay phase or serializing epoch changes;
- domain conflicts can squash writeback instead of leaking mismatched state.

Determinism here does not mean "no runtime choice exists." It means runtime choice is constrained, classified, and replay-aware.

## Replay-sensitive behavior

Replay-sensitive state appears in multiple layers:

- loop-buffer replay phase context
- stable donor mask
- previous-lane reuse hints
- replay certificate and class-template reuse
- serializing epoch invalidation

`SetReplayPhaseContext(...)` in the scheduler synchronizes replay-phase changes and clears reuse state when epochs change or replay deactivates.

## Retire contour

Retirement is explicit and evidence-bearing:

- lane-generated effects are captured into retire batches;
- serializing boundaries are noted from `SerializationClass.FullSerial` and `SerializationClass.VmxSerial`;
- trace events are recorded on retirement;
- cross-thread densification-injected memory work can release speculation budget on retirement;
- retire visibility contour certificates are published.

This reinforces the architecture's "evidence and legality are first-class" design style.

## Current limitations

- The WhiteBook does not claim complete hardware timing equivalence beyond the emulator/runtime implementation.
- Stream-control and VMX surfaces are guarded; not every contour is always wired in every runtime configuration.
- Foreground issue and full-bundle decode coexist, which makes the contour explicit but also more complex than a simple scalar pipeline description.

## Open questions / future work

- Whether the canonical/derived issue-plan separation should be surfaced as a formal public contract.
- Whether pipeline contour certificates should be exported in a more externalized evidence format.
