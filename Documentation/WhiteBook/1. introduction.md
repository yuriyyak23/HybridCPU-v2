# Introduction

## Goal of this chapter

This chapter defines the scope, evidence model, and documentation method for the HybridCPU ISE WhiteBook.

## Scope

This WhiteBook documents the current architecture embodied by the repository at `HybridCPU_ISE/`, not an idealized or historical design. It is written as an architecture reference for engineers who need to understand how the emulator/runtime currently behaves, what the compiler is expected to provide, and where the implementation remains intentionally incomplete.

The intended document type is closer to an architecture whitebook or design reference than to product messaging. Claims are included only when they are backed by code, tests, or both.

## Method

The WhiteBook was assembled from three input classes:

1. `Documentation/WhiteBook/README.md` as the initial narrative frame.
2. Current runtime code under `HybridCPU_ISE/Arch/`, `HybridCPU_ISE/Core/`, and `HybridCPU_ISE/Processor/`.
3. Current test suites under `HybridCPU_ISE.Tests/tests/`.
4. The instruction-refactor closure documents under `Documentation/InstructionsRefactor/WhiteBook/`, which summarize the completed ISA-surface phases and their runtime-owned evidence boundaries.
5. Specialized authority packs under `Documentation/Virtualization WhiteBook/`, `Documentation/Stream WhiteBook/`, and `Documentation/SecureCompute WhiteBook/`.

When README and code diverge, code is treated as authoritative. When code and tests diverge from older documentation paths, the live repository layout is treated as authoritative.

## Documentation principles

The WhiteBook uses four evidence labels throughout:

- Confirmed in code: behavior directly expressed by live implementation.
- Confirmed by tests: behavior exercised by named test surfaces.
- Current limitation: behavior explicitly absent, partial, or blocked by configuration.
- Open question / future work: behavior suggested by scaffolding or research comments but not yet mainline.

The document is intentionally fail-closed. If a feature cannot be confirmed from live code, it is not presented as implemented.

Specialized WhiteBooks are used as domain authority packs, but the same rule still applies: when specialized prose, older phase text, and live code disagree, this base WhiteBook labels the difference as implemented behavior, denied/fail-closed behavior, model-only/helper-only surface, compatibility projection, future work, or test/conformance evidence instead of flattening the distinction.

## Current architectural thesis

The current codebase implements a replay-stable SMT-VLIW emulator/runtime within the current evidence envelope, with:

- a fixed 8-slot bundle carrier;
- a fixed 4-way SMT core model;
- a typed-slot scheduling discipline with class-level admission and deterministic late lane binding;
- a replay-aware densification path;
- an explicit legality layer built around certificates, guard planes, and domain checks;
- an assist plane that is architecturally invisible but operationally explicit;
- a closed instruction-runtime surface for the currently published scalar, branch/control, load/store, atomic, and bounded fence contours;
- explicit non-executable support categories for descriptor-only, carrier-only, parser-only, optional-disabled, and reserved instruction rows;
- telemetry and replay evidence surfaces that are treated as part of the architecture, not just as debug output.
- neutral virtualization, stream/lane6/lane7, and SecureCompute boundaries that are now documented in specialized WhiteBooks rather than expanded fully in this base document.

At the same time, the implementation is not a generic out-of-order machine, not a free-form superscalar scheduler, and not a complete production attestation stack.

Determinism claims in this WhiteBook are therefore bounded: they refer to replay-aware lane choice, fail-closed boundaries, and targeted evidence suites, not to a universal theorem over every possible runtime configuration.

The same bounded-language rule applies to the instruction set. The current ISA refactor closes scalar repair through `ZEXT.W`, canonical branch/control transport, typed scalar load/store, LR/SC and AMO W/D retire semantics, atomic acquire/release ordering, and canonical zero-payload `FENCE` / `FENCE_I`. It does not claim a global cache/TLB/DMA/coherence theorem, broad lane6/lane7 device coverage beyond the current gated contours, executable gather/scatter, or compiler emission for every future contour.

The same bounded-language rule applies to specialized domains:

- Virtualization authority is neutral runtime authority; VMX is compatibility ABI/projection vocabulary and denied/fail-closed vocabulary, not the backend owner.
- Stream and accelerator carriers separate typed sideband, token/helper/model APIs, commit authority, and pipeline execution. Evidence surfaces such as telemetry, replay, certificates, tokens, helper observations, and backend diagnostics do not grant authority.
- SecureCompute is a neutral descriptor/admission discipline. It is not VMX mode, secure VMCS, `VmxCaps` authority, CHERI ISA, tagged memory, or a completed secure backend runtime.
- Compiler-visible typed-slot metadata mirrors runtime legality and placement, but it does not replace them.

## How to read the WhiteBook

Readers who need the binary and front-end contract should begin with `vliw-binary-format.md`. Readers who need the runtime scheduling story should continue with `architectural-model.md`, `execution-model.md`, and `typed-slot-scheduling.md`.
