# Current State and Modernization Tracks

## Goal of this chapter

This chapter separates what is already landed in the current mainline runtime from what remains partially landed, optional, or research-oriented.

## Mainline state

The strongest current architectural result in the repository is the typed-slot runtime path over the fixed typed `W=8` machine. This includes:

- explicit `SlotClass` taxonomy;
- class-capacity accounting;
- two-stage typed-slot admission (`TryClassAdmission` then `TryMaterializeLane`);
- deterministic late lane binding;
- replay-aware stable-donor lane reuse;
- legality authority routed through explicit `LegalityDecision` values;
- telemetry for inject/reject outcomes;
- compiler/runtime typed-slot agreement checks.

This is not just an idea documented in README. The corresponding classes and runtime paths are present in the live scheduler and safety layers.

## Canonical topology reminder

All modernization tracks in this chapter sit on the same canonical lane map:

- `ALU[0..3]`
- `LSU[4..5]`
- `DMA/stream[6]`
- `Branch/System[7]`

The lane-7 branch/system alias is part of the active scheduling model, not an incidental implementation detail.

## Active frontend boundary

The active frontend is native VLIW only. Compatibility ingress remains quarantined to compat or diagnostic seams, and DBT or scalar-generalized decode are not active proof surfaces in the current repository.

Boundary evidence is already present in the live suite surface:

- `Phase11NativeVliwDiagnosticDecodeTests.cs`
- `Phase12VliwCompatFreezeTests.cs`
- `Phase12RetiredCompatPolicyBitBoundaryTests.cs`

## Track A: Typed-slot runtime modernization

The following runtime capabilities are concretely present in code:

- `SlotClassDefinitions.cs`: topology, pinning, reject taxonomy, bundle facts
- `SlotClassCapacity.cs`: occupancy and remaining-capacity model
- `MicroOpScheduler.Admission.cs`: class admission plus deterministic lane materialization
- `DeterministicLaneChooser.cs`: lowest-free plus replay-hint selection
- `MicroOpScheduler.SMT.cs`: typed-slot path integrated into live intra-core SMT packing
- `SafetyVerifier.TypedSlot.cs`: fact validation and agreement checks

The architectural shift is from "fill an exact empty physical slot if possible" to "admit into class capacity, then bind a legal lane deterministically."

## Track B: Execution evidence and replay-oriented modernization

The second strong line of implementation is the evidence-oriented execution complex:

- loop-buffer replay phase context;
- replay-aware legality reuse;
- phase-certified bundle packing;
- boundary guards and serializing invalidation;
- determinism and lane-reuse telemetry;
- trace and telemetry export;
- replay token capture/rollback substrate.

This track strengthens the scheduler and pipeline without turning the system into a nondeterministic hidden scheduler.

## Backend truthfulness boundary

The live backend retains explicit rename/commit state. Current runtime state explicitly includes:

- `PhysicalRegisterFile`
- `RenameMap`
- `CommitMap`
- `FreeList`

That backend substrate coexists with legality, certificate, and typed-slot machinery. The current architectural claim is therefore not "rename elimination," but rather that bundle legality and certificate-governed admission are made explicit and analyzable instead of being buried inside a hidden scheduler policy surface.

## What is landed but still optional or guarded

Some mechanisms exist in code but are clearly optional, conditional, or exploratory:

- `EnableShadowOracle` instrumentation in the scheduler
- pipelined FSP mode
- optional matrix and scalar residual opcode contours, which are rejected in canonical decode
- execution-surface guards for stream control and VMX routing when surfaces are not wired

These must not be described as unconditional production behavior.

## What is not yet fully landed end-to-end

A key boundary remains between runtime readiness and compiler adoption:

- runtime code contains typed-slot facts and agreement checks;
- runtime comments explicitly say correct execution does not require non-default `SlotMetadata`;
- the compiler-side story is not yet "compiler facts are mandatory for correctness."

That means the runtime modernization is ahead of the full compiler-mainline adoption story.

## Current limitations

- Full compiler-side typed-slot fact emission is not yet required for runtime correctness.
- Some modernization lines are present as guarded or exploratory instrumentation rather than as the sole runtime authority.
- Historical notes remain in the tree, but they are explicitly non-authoritative and should not outrank the current README or WhiteBook entry points.

## Open questions / future work

- When and how typed-slot facts become default compiler output rather than optional or diagnostic transport.
- Whether shadow-oracle instrumentation graduates into a stronger validation or tuning workflow.
- Whether the repository should collapse more repository-facing summaries onto the same small set of authoritative architecture and validation entry points.
