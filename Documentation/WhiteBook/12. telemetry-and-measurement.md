# Telemetry and Measurement

## Goal of this chapter

This chapter describes telemetry as an architectural evidence surface. The purpose is not merely debugging, but measuring whether typed-slot scheduling, replay-aware packing, and legality policy behave as designed.

## Telemetry is part of the architecture

In the current codebase, telemetry is tightly coupled to architectural mechanisms:

- typed-slot admission
- legality rejection
- replay and class-template reuse
- fairness and starvation
- bank pressure and memory hazards
- retired width and utilization

This is why the WhiteBook treats telemetry as architecture evidence, not as incidental debug noise.

## Core telemetry sources

The main runtime telemetry surfaces are:

- `TelemetryCounters`
- `SchedulerPhaseMetrics`
- `TelemetryExporter`
- `TypedSlotTelemetryProfile`
- `TelemetryProfileReader`
- `LoopPhaseClassProfile`
- trace sinks and replay evidence summaries

These surfaces together cover both direct counters and derived profiles.

## What is measured

Current exported or derivable measurements include:

- injections per slot class
- rejects per slot class
- rejects by `TypedSlotRejectReason`
- assist quota and assist backpressure rejects
- replay template hits and misses
- replay invalidation reasons and certificate reuse invalidations
- fairness starvation events
- per-VT injection and rejection counts
- certificate-pressure breakdown
- bank-pending rejects per bank
- hazard-type counts
- cross-domain reject counts
- eligibility-mask telemetry
- loop-phase class profiles
- scalar and non-scalar retired lane counts
- average retired width
- NOP density and bundle utilization

This coverage is much richer than a basic instruction-retired counter set.

## Trace-coupled counters

`TelemetryCounters` also tracks global and per-VT execution counters from trace events, including:

- cycles
- bundles retired
- instructions retired
- VM exits
- barriers
- FSP steal count
- replay count
- atomic and synchronization counters per VT

That ties the architectural trace model directly into measurable runtime state.

## Export model

`TelemetryExporter.BuildProfile(...)` turns scheduler and pipeline state into a `TypedSlotTelemetryProfile` JSON-ready artifact.

The profile is not just a flat bag of counters. It preserves typed architecture vocabulary:

- `SlotClass`
- `TypedSlotRejectReason`
- legality authority details
- loop-phase replay profiles
- certificate-pressure breakdown

The exported reject vocabulary is expected to stay aligned with runtime legality and assist semantics, including:

- `TypedSlotRejectReason.AssistQuotaReject`
- `TypedSlotRejectReason.AssistBackpressureReject`
- replay invalidation and phase-certificate invalidation diagnostics carried through replay evidence surfaces

Compiler-side telemetry consumption is intentionally keyed to the same vocabulary.
`TelemetryProfileReader.GetRejectRate(...)` reads reason-specific reject shares from `TypedSlotTelemetryProfile.RejectsByReason`, so exported runtime evidence and compiler-side interpretation have to use the same reject names.

`TraceSink` and `ReplayEngine.Analysis` extend that evidence path with replay invalidation, certificate reuse, determinism-envelope, and assist telemetry summaries rather than reducing everything to a single throughput counter.

This supports the README's intent that telemetry feed back into compiler/runtime understanding, but with current code-backed types rather than aspirational prose.

## Evidence use

The measurement philosophy visible in code is:

- instrument structural outcomes, not only final throughput;
- keep reject reasons explicit;
- expose replay and determinism-tax signals;
- preserve per-VT and per-class visibility where helpful;
- export profiles suitable for offline analysis.

## Current limitations

- The telemetry schema is rich, but it is still source-defined rather than frozen as a long-term external compatibility contract.
- Some exported fields are optional and appear only when the relevant runtime surface was active.
- The current WhiteBook does not claim a finished external dashboarding or artifact-management workflow.

## Open questions / future work

- Stable versioning of telemetry schema.
- Better long-run correlation of replay evidence, legality evidence, and performance evidence.
- Clearer separation between developer diagnostics and publishable evidence artifacts.
