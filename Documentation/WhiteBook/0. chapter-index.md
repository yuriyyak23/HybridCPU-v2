## Goal of this chapter

This chapter provides the entry map for the WhiteBook and explains how the chapters relate to the current HybridCPU ISE codebase.

The supporting chapter `19. references-and-reading-order.md` lists the main code and test surfaces that back the WhiteBook. The focused artifact `Documentation/operational-semantics.md` externalizes the state tuple and cycle-step relation for the active runtime, `20. legality-predicate.md` externalizes the canonical Stage A / Stage B legality decomposition for the active typed-slot SMT path, `23. certificate-semantics-and-legality-evidence.md` states the certificate/structural-identity evidence chain, `21. reject-taxonomy-matrix.md` ties compiler preflight categories to runtime reject families without inventing a false one-to-one enum, and `22. typed-slot-contract-staging.md` freezes the current `ValidationOnly` status of typed-slot facts.

The instruction-refactor WhiteBook under `Documentation/InstructionsRefactor/WhiteBook/` is the ISA-surface closure overlay for this base WhiteBook. It summarizes the current bounded ISA contour: scalar repair through `ZEXT.W`, branch/control target transport, scalar load/store, LR/SC and AMO W/D retire semantics, acquire/release ordering, canonical zero-payload `FENCE` / `FENCE_I`, and explicit non-executable/future-gated rows.

## Specialized WhiteBook reading path

This base WhiteBook is the top-level architectural map. The specialized WhiteBooks are authority packs for deeper domain details and should be read after the base chapters identify the relevant boundary:

- `Documentation/Virtualization WhiteBook/`: VMX compatibility freeze, neutral runtime virtualization authority, VMCS/VMCSv2 projection vocabulary, VMREAD admitted/denied fields, VMCALL trap/backend-admission fences, and VMX/SecureCompute denial boundaries.
- `Documentation/Stream WhiteBook/`: canonical execution-plane taxonomy for
  VectorStream (`StreamEngine`, SRF, `BurstIO`, `VectorALU`), MatrixTile memory
  and compute, lane6 DSC, assists, and lane7 L7-SDC.
- `Documentation/SecureCompute WhiteBook/SecureCompute HybridCPU-v2 WhiteBook.md`: neutral SecureCompute descriptor/admission discipline, no-effect disabled semantics, secure memory/measurement/evidence/migration/I/O/hypercall policies, Layer 2 grant discipline, VMX compatibility fences, and open backend-execution decisions.

Topic-oriented path:

- For virtualization authority, read this index, `3. system-overview.md`, `10. safety-isolation-and-legality.md`, then `Virtualization WhiteBook/01_Executive_Summary.md` and `Virtualization WhiteBook/16_Current_State_And_Closure_Matrix.md`.
- For stream and matrix questions, read `4. architectural-model.md`,
  `5. execution-model.md`, then `Stream WhiteBook/00_README.md`,
  `Stream WhiteBook/02_VectorStream/00_README.md`, and
  `Stream WhiteBook/03_MatrixTile/00_README.md`.
- For lane6/lane7 questions, continue with
  `Stream WhiteBook/05_Lane6_Lane7_Separation.md`,
  `Stream WhiteBook/DmaStreamCompute/01_Current_Contract.md`, and
  `Stream WhiteBook/ExternalAccelerators/01_L7_SDC_Executive_Summary.md`.
- For SecureCompute authority, read `10. safety-isolation-and-legality.md`, `15. architectural-boundaries-and-non-goals.md`, then the SecureCompute WhiteBook sections 1, 5, 11, 13, 14, and 17.
- For validation, read `13. validation-status-and-test-evidence.md` and the source-reference chapters in each specialized WhiteBook.

## What is authoritative in code

The chapter split is derived from the current runtime, compiler contract, and test tree under:

- `HybridCPU_ISE/Arch/`
- `HybridCPU_ISE/Core/`
- `HybridCPU_ISE/Processor/`
- `HybridCPU_ISE.Tests/tests/`
- `Documentation/InstructionsRefactor/WhiteBook/`

The existing `Documentation/WhiteBook/README.md` is used as a narrative starting point, but not as the final authority.
