## Goal of this chapter

This chapter provides the entry map for the WhiteBook and explains how the chapters relate to the current HybridCPU ISE codebase.

## Chapter map

- `1. introduction.md`: scope, method, and fail-closed documentation rules.
- `2. vliw-binary-format.md`: the authoritative slot and bundle container format.
- `3. system-overview.md`: repository-level component map and runtime relationship model.
- `17. current-state-and-modernization-tracks.md`: what is already landed versus exploratory.
- `4. architectural-model.md`: scalar, VLIW, streaming-vector, and 4-way SMT model.
- `5. execution-model.md`: fetch, decode, issue, materialization, memory, and retire contours.
- `Documentation/operational-semantics.md`: explicit machine-state tuple, cycle-step relation, and bounded sub-relations for legality, replay, retire, memory, exception, rollback, and backend truthfulness.
- `6. typed-slot-scheduling.md`: slot classes, pinning, admission, lane binding, and reject taxonomy.
- `7. compiler-runtime-contract.md`: versioned contract, metadata transport, and runtime validation.
- `8. smt-vliw-packing-and-densification.md`: heterogeneous 8-wide issue and replay-aware SMT packing.
- `9. assist-runtime-donor-taxonomy-and-carriers.md`: donor taxonomy, assist kinds, and carrier planes.
- `10. safety-isolation-and-legality.md`: certificates, domain guards, and legality authority sources.
- `20. legality-predicate.md`: canonical legality decomposition tying Stage A admission, guard-plane precedence, replay/certificate authority, and Stage B lane materialization together.
- `23. certificate-semantics-and-legality-evidence.md`: paper-facing descriptor, certificate structural identity, replay reuse key, and legality-evidence diagram.
- `21. reject-taxonomy-matrix.md`: compiler preflight classes, runtime reject families, verifier classification flags, and retained telemetry buckets.
- `22. typed-slot-contract-staging.md`: staged status of typed-slot facts, current `ValidationOnly` mode, and the future escalation ladder.
- `11. replay-determinism-and-evidence.md`: replay phase model, determinism envelope, and evidence surfaces.
- `12. telemetry-and-measurement.md`: measurement as architectural evidence.
- `13. validation-status-and-test-evidence.md`: test landscape and explicit evidence boundaries.
- `14. repository-layout-and-code-map.md`: live code map and stale-document path mismatches.
- `15. architectural-boundaries-and-non-goals.md`: what the current system is not.
- `16. open-research-directions.md`: research-grade or still-open lines of work.
- `18. conclusion.md`: synthesis of the current architectural state.

## What is authoritative in code

The chapter split is derived from the current runtime, compiler contract, and test tree under:

- `HybridCPU_ISE/Arch/`
- `HybridCPU_ISE/Core/`
- `HybridCPU_ISE/Processor/`
- `HybridCPU_ISE.Tests/tests/`

The existing `Documentation/WhiteBook/README.md` is used as a narrative starting point, but not as the final authority.

## Current limitations

This index is a navigation aid, not a normative spec by itself. Readers should use the per-chapter authority sections for concrete claims.
