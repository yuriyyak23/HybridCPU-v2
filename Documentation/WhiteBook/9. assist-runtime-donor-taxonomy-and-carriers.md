# Assist Runtime, Donor Taxonomy, and Carriers

## Goal of this chapter

This chapter explains the current assist plane as implemented: assist kinds, donor taxonomy, carrier planes, lane-6 DMA versus LSU-hosted execution, and the boundary between donor provenance and bundle densification.

## Assist plane overview

`AssistMicroOp` is the main assist carrier type. Current code defines assists as:

- architecturally invisible
- non-retiring
- replay-discardable
- memory-oriented
- placement-bound and legality-bound like ordinary micro-ops

This is an explicit runtime plane, not an informal prefetch heuristic hidden behind ordinary loads.
It is also a bounded invisibility claim: assist is architecturally invisible and
non-retiring, but it remains boundedly observable through carrier-memory effects
plus replay and telemetry evidence. The reviewer-facing closure surface for that
visibility envelope lives in `HybridCPU_ISE/docs/assist-semantics.md`.

## Orthogonal axes

The implementation keeps three concepts separate:

1. Bundle densification: who gets a slot in a bundle this cycle
2. Donor taxonomy: where the assist's data provenance comes from
3. Carrier plane: which physical execution class runs the assist

This separation is important for accurate architecture writing.

## Landed assist kinds and modes

Current runtime enums define:

- `AssistKind`: `DonorPrefetch`, `Ldsa`, `Vdsa`
- `AssistExecutionMode`: `CachePrefetch`, `StreamRegisterPrefetch`
- `AssistCarrierKind`: `LsuHosted`, `Lane6Dma`

Carrier-plane selection is explicit. It is not inferred indirectly from donor identity.

## Carrier semantics

The current carrier-to-slot-class mapping is:

- `LsuHosted` -> `SlotClass.LsuClass`
- `Lane6Dma` -> `SlotClass.DmaStreamClass`

`AssistTupleSupport.IsSupportedCarrierTuple(...)` constrains legal combinations:

- `Vdsa` is only legal as `StreamRegisterPrefetch + Lane6Dma`
- `Ldsa` and `DonorPrefetch` may use either:
  - `CachePrefetch + LsuHosted`
  - `StreamRegisterPrefetch + Lane6Dma`

So lane 6 is not a generic "assist lane" in abstract. It is the explicit DMA/stream carrier used for specific assist contours.

## Donor taxonomy

`AssistDonorSourceKind` distinguishes:

- same-thread local seed reuse
- intra-core cross-VT vector donor
- inter-core same-VT vector or scalar seed
- inter-core cross-VT donor seed
- inter-core vector writeback variants
- inter-core hot-load, hot-store, cold-store, and default-store scalar variants

This taxonomy answers the question "who donates the data contour," not "who owns the bundle slot."

## Current seed classification rules

`AssistMicroOp.TryClassifySeed(...)` and inter-core classification logic implement a few important current rules:

- vector seeds map toward `Vdsa`, stream-register prefetch, and lane-6 DMA
- local scalar assists generally remain LSU-hosted cache-prefetch style
- widened lane-6 scalar assists mainly appear in inter-core paths
- scalar store assist classification differentiates hot, cold, and default store cases in inter-core transport

## Current local default path

The default local path `AssistMicroOp.TryCreateFromSeed(...)` normalizes the donor VT from the seed and defaults the target VT to the same donor VT unless a different target is explicitly supplied.

This matters for correctness of architectural claims:

- same-VT local assists are mainline behavior;
- intra-core cross-VT donor taxonomy exists in code;
- but default local assist creation does not automatically create cross-VT donor assists.

## Legality matrix

`AssistDonorSourceDescriptor.IsLegalFor(...)` is the main donor/carrier legality matrix.

The reviewer-facing frozen matrix now lives in `HybridCPU_ISE/docs/assist-semantics.md`.
This chapter remains the narrative summary; the phase artifact is the exact four-tuple closure surface.

Important current boundaries:

- intra-core cross-VT donor assists are only legal for `Vdsa`
- `LsuHosted` scalar assists are same-thread only
- lane-6 scalar assists are tied to inter-core contours
- donor/core/pod ownership must remain consistent with the selected donor class

This means the donor taxonomy is not just descriptive. It is an enforceable legality surface.

## Quota and backpressure

The assist plane also has explicit resource policy:

- `AssistMemoryQuota`
- `AssistBackpressurePolicy`
- `AssistCachePartitionPolicy`

`MicroOpScheduler.Assist.cs` enforces:

- assist bundle quota
- assist-only quota reservation
- assist-specific backpressure reservation
- typed-slot placement for assist candidates

Current code limits assist injection to at most one assist per bundle through `AssistBundleQuota = 1`.
The exact Phase 04 quota/backpressure closure lives in `HybridCPU_ISE/docs/assist-semantics.md`.
It keeps `AssistQuotaRejectKind.IssueCredits`, `AssistQuotaRejectKind.LineCredits`,
`AssistBackpressureRejectKind.SharedOuterCap`, `AssistBackpressureRejectKind.OutstandingMemory`,
and `AssistBackpressureRejectKind.DmaStreamRegisterFile` as admission-failure evidence rather
than architectural retire-visible effects.

## Inter-core transport

Inter-core assist paths use explicit transport objects and ownership snapshots rather than implicit remote access. The inter-core injection path:

- nominates transport from a donor core;
- checks staleness and owner/domain consistency;
- classifies the assist from transport;
- routes it through legality and lane materialization;
- applies assist quota and backpressure checks;
- commits the assist as a runtime-local invisible micro-op.

## Current limitations

- Default local assist creation does not automatically generate intra-core cross-VT donor assists.
- Scalar lane-6 widening is mainly an inter-core contour, not the default local scalar assist behavior.
- Assists are architecturally invisible and non-retiring; they are not a retire-visible ISA-level feature.

## Open questions / future work

- Whether the runtime should generate more intra-core cross-VT donor assists automatically.
- Whether assist quota and carrier policy should become more workload-adaptive.
- Whether the assist taxonomy should eventually be surfaced as a more formal public subsystem spec.
