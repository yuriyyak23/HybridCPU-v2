# Reject Taxonomy Matrix

## Goal of this chapter

This chapter closes the current reject-taxonomy story across compiler preflight, runtime scheduler rejects, verifier-side classification, and retained telemetry buckets. It is intentionally not a claim that every layer uses one shared enum.

## Canonical surfaces

The active surfaces are:

- compiler preflight: `AdmissibilityClassification`
- compiler/runtime relation helper: `AdmissibilityRuntimeVocabulary`
- runtime scheduler reject code: `TypedSlotRejectReason`
- runtime classifier: `TypedSlotRejectClassification`
- fine-grained checker decision: `LegalityDecision` with `RejectKind`, `CertificateRejectDetail`, and `LegalityAuthoritySource`

The closure rule is simple: these surfaces must be read as one matrix, not as one flattened vocabulary.
`AdmissibilityRuntimeVocabulary` is the canonical code seam that says which compiler-side categories have a proven direct scheduler twin, which remain compiler-only structural invalidities, and which are only compatibility-adjacent to runtime families.

## Matrix

| Family | Compiler preflight surface | Runtime scheduler surface | Verifier / diagnostic surface | Current contract |
|---|---|---|---|---|
| Static class-capacity overcommit | `TypedSlotClassCapacityExceeded` | `StaticClassOvercommit` | `IsClassCapacityIssue=true`, `IsStaticOvercommit=true` | Closest direct structural mapping. |
| Dynamic class-capacity exhaustion | `StructurallyAdmissible` can still reach runtime | `DynamicClassExhaustion` | `IsClassCapacityIssue=true`, `IsDynamicExhaustion=true`, `IsDynamicStateIssue=true` | Runtime-only outcome after live occupancy changes during the pass. |
| Certificate or legality-service denial | `StructurallyAdmissible` or retained `SafetyMaskConflict` preflight | `ResourceConflict` | `CertificateRejectDetail`, `_lastSmtLegalityRejectKind`, `_lastSmtLegalityAuthoritySource` | Active Stage A currently collapses checker-owned denials to `ResourceConflict` at the scheduler-visible reject layer. |
| Dynamic state pressure | `StructurallyAdmissible` | `ScoreboardReject`, `BankPendingReject`, `HardwareBudgetReject`, `SpeculationBudgetReject`, `AssistQuotaReject`, `AssistBackpressureReject` | `IsDynamicStateIssue=true` | Runtime-only family after structural preflight has already passed. |
| Stage B lane materialization | `StructurallyAdmissible` | `PinnedLaneConflict`, `LateBindingConflict` | `IsPinnedConflict` for the pinned case | Lane-materialization failures after Stage A success. |
| Compile-time aliased lane invalidity | `TypedSlotAliasedLaneConflict` | no dedicated runtime reject twin | no direct `TypedSlotRejectClassification` twin | Compiler-only structural invalidity; must fail before active mainline. |
| Compile-time invalid typed-slot facts | `TypedSlotFactsInvalid` | no dedicated runtime reject twin | `ValidateTypedSlotFacts(...)` / `VerifyTypedSlotAgreement(...)` remain the relevant surfaces | Compiler/runtime agreement failure, not a live dynamic reject family. |
| Retained domain bucket | none in compiler preflight | `DomainReject` remains in `TypedSlotRejectReason` | retained telemetry and helper surfaces | Retained bucket exists, but active `TryClassAdmission(...)` legality-service denials currently collapse to `ResourceConflict` instead of emitting `DomainReject` directly. |

## Key closure rules

- `StructurallyAdmissible` means compiler preflight passed. It is not a guarantee that runtime will avoid later dynamic or legality-service rejects.
- `TypedSlotAliasedLaneConflict` and `TypedSlotFactsInvalid` are compile-time-only categories. They should not be described as active runtime reject families.
- Stage A legality-service denials are scheduler-visible as `ResourceConflict`, while the finer checker reason remains in `RejectKind` and `LegalityAuthoritySource`.
- `SafetyMaskConflict` is a secondary compatibility-era structural diagnostic. It is adjacent to runtime resource-conflict reasoning, but it is not the authoritative typed-slot mainline contract.
- Only `TypedSlotClassCapacityExceeded -> StaticClassOvercommit` is published as a direct structural twin in code today. Everything else must stay explicitly marked as runtime-continuation, compatibility-adjacent, or compiler-only.

## Current limitations

- The matrix remains intentionally multi-layered across compiler, scheduler, and verifier surfaces; `AdmissibilityRuntimeVocabulary` only closes the compiler/runtime relation seam and does not replace the underlying runtime taxonomies.
- `DomainReject` remains a retained taxonomy bucket even though the active legality-service path preserves finer guard reasons in `LegalityDecision` and collapses the scheduler-visible reject to `ResourceConflict`.
- `SafetyMaskConflict` still exists as a secondary compiler diagnostic even though typed-slot structural checks are the primary handoff.

## Open questions / future work

- Decide whether a future runtime-facing reject layer should expose more of `RejectKind` directly instead of collapsing legality-service denials to `ResourceConflict`.
- Determine whether retained buckets such as `DomainReject` should remain as telemetry compatibility surfaces or be further quarantined.
- Generate the matrix automatically from source annotations once the Phase 02 closure is complete.
