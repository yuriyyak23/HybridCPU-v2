# Typed-Slot Scheduling

## Goal of this chapter

This chapter defines the current typed-slot scheduling model: slot classes, lane pinning, class-level admission, deterministic late binding, reject taxonomy, and runtime authority boundaries.

## Slot classes and lane map

The runtime defines a fixed `SlotClass` taxonomy:

- `AluClass`
- `LsuClass`
- `DmaStreamClass`
- `BranchControl`
- `SystemSingleton`
- `Unclassified`

`SlotClassLaneMap` fixes the lane masks:

- `AluClass` -> lanes 0..3
- `LsuClass` -> lanes 4..5
- `DmaStreamClass` -> lane 6
- `BranchControl` -> lane 7
- `SystemSingleton` -> lane 7

`BranchControl` and `SystemSingleton` are explicitly aliased on the same physical lane.

## Pinning model

Each operation is either:

- `ClassFlexible`
- `HardPinned`

`HardPinned` means the runtime must place the operation in `PinnedLaneId`. `ClassFlexible` means the operation is admitted at class level first, then placed into one currently legal lane of that class.

## Class admission

Stage A of the typed-slot pipeline is `TryClassAdmission(...)`. It checks, in current code order:

1. class-capacity availability
2. legality-service decision
3. outer-cap dynamic gates

The dedicated artifact `20. legality-predicate.md` freezes this as the canonical
Stage A contract. Guard-plane checks, replay/template reuse, and structural
witness fallback are Stage A substeps inside the legality-service decision;
they are not a separate Stage B or post-admission repair path.

The outer-cap gates include:

- scoreboard pressure
- bank-pending rejection
- hardware memory budget
- speculation budget
- assist-only quota and backpressure for assists

This makes current typed-slot scheduling more than a static class-count check. The class model is structural, but live authority is runtime and state-sensitive.

## Late lane binding

Stage B is `TryMaterializeLane(...)`:

- hard-pinned candidates go directly to the specified lane if free;
- class-flexible candidates intersect class mask with currently free lanes;
- replay-aware stable-donor masking may further restrict the eligible lanes;
- `DeterministicLaneChooser.SelectWithReplayHint(...)` selects a lane.

The deterministic chooser prefers:

- replay-lane reuse if replay is active and the previous lane is still legal;
- otherwise the lowest free lane.

This is the core reason the model should be described as "late binding" rather than as exact-slot preassignment.
Stage B does not reopen legality. It only materializes one lane from the already admitted class/topology envelope.

## Compiler-facing structural facts

The shared vocabulary for compiler/runtime structural agreement is `TypedSlotBundleFacts`, which contains:

- per-slot required class
- pinning mask
- counts per class
- pinned and flexible totals

`BundleLegalityAnalyzer` derives canonical typed-slot facts from decoded bundles. `SafetyVerifier.ValidateTypedSlotFacts(...)` and `VerifyTypedSlotAgreement(...)` check consistency.
The dedicated artifact `22. typed-slot-contract-staging.md` freezes the staged status of these facts: current mainline mode is `ValidationOnly`, so missing facts do not yet invalidate canonical execution by themselves.

## Reject taxonomy

The reject model is explicit. Current runtime reject reasons include:

- `StaticClassOvercommit`
- `DynamicClassExhaustion`
- `ResourceConflict`
- `DomainReject`
- `ScoreboardReject`
- `BankPendingReject`
- `HardwareBudgetReject`
- `SpeculationBudgetReject`
- `AssistQuotaReject`
- `AssistBackpressureReject`
- `PinnedLaneConflict`
- `LateBindingConflict`
- `FairnessDeferred`

This reject taxonomy is important because it makes scheduler behavior inspectable. "Cannot inject" is not a single opaque outcome.

## Runtime authority

The runtime remains authoritative for:

- live capacity state;
- legality decisions;
- owner/domain guard behavior;
- replay-aware admissibility;
- lane choice;
- dynamic rejection.

The compiler can describe structure and intent, but it does not override runtime legality. This is a central design principle of the current implementation.

## Current limitations

- `Unclassified` still exists as a compatibility or placeholder path.
- Branch and system-singleton work contend for one aliased lane.
- Typed-slot facts are validated and consumed by runtime, but not yet a mandatory correctness dependency for canonical execution.

## Open questions / future work

- Whether more fine-grained class families are needed.
- Whether branch/system aliasing on lane 7 should remain permanent.
- Whether the compiler should eventually emit richer per-bundle facts by default.
