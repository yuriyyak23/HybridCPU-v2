# Typed-Slot Contract Staging

## Goal of this chapter

This chapter freezes the current staged status of compiler-emitted typed-slot facts so the repository can describe their role truthfully without implying that they are already a mandatory correctness substrate.

## Canonical staging surface

The canonical code surface for this story is:

- `TypedSlotFactMode`
- `TypedSlotFactStaging.CurrentMode`
- `CompilerTypedSlotPolicy`
- `CompilerContract.CurrentTypedSlotPolicy`

Current mainline status is:

- `TypedSlotFactStaging.CurrentMode == TypedSlotFactMode.ValidationOnly`
- `CompilerContract.CurrentTypedSlotPolicy.Mode == CompilerTypedSlotPolicyMode.CompatibilityValidation`

That means the repository has an explicit staged vocabulary, but only the weakest stage is active today.

## Current mainline meaning: `ValidationOnly`

`ValidationOnly` means:

- compiler and runtime may emit, transport, and validate `TypedSlotBundleFacts`;
- `ValidateTypedSlotFacts(...)`, `VerifyTypedSlotAgreement(...)`, and `HybridCpuTypedSlotFactsEmitter.ValidateEmittedFacts(...)` are active agreement surfaces;
- compiler preflight may still classify a bundle as structurally invalid when emitted facts contradict typed-slot topology;
- runtime legality remains the final authority for canonical execution;
- missing or default typed-slot facts are still accepted on the canonical runtime path.

The key truth constraint is simple: typed-slot facts are landed validation and agreement surfaces, but they are not yet the sole correctness carrier for typed-slot admission.

## Runtime bridge policy: `CompatibilityValidation`

`CompilerTypedSlotPolicy` is the bridge-level policy that makes the current missing-facts and disagreement behavior explicit:

- missing or default typed-slot facts are accepted as compatible input;
- present facts are validated when the bridge has a runtime bundle and validation is enabled;
- structural agreement failures are recorded as `AgreementViolationReport` evidence and quarantine-logged through `CompilerTypedSlotIngressAction.QuarantinedAgreementFailure`;
- compatibility validation does not reject canonical execution solely because compiler facts disagree with runtime validation;
- validation-disabled or no-bundle compatibility ingress records the facts without treating them as authority.

This is the current mainline mode. It is intentionally weaker than fail-closed admission because the repository does not yet prove that typed-slot facts are mandatory correctness substrate.

## Optional strict verification mode

`CompilerTypedSlotPolicy.StrictVerification` is a selectable proof mode for surfaces that need stricter verification:

- missing facts remain compatible, so strict verification is not the future required-facts mode;
- present facts must be verifiable against a runtime bundle;
- structural agreement failures are recorded and then rejected;
- dynamic runtime rejects remain runtime-owned legality outcomes rather than compiler-fact structural mismatches.

Strict verification is therefore a bridge verification mode, not a new ISA rule and not a claim that all compiler ingress already requires typed-slot facts.

## Staged evolution path

The repository now reserves the following staged ladder:

1. `ValidationOnly`
   Facts are validated when present. Missing facts remain compatible with canonical execution.
2. `WarnOnMissing`
   Future mode where missing facts would produce explicit warnings or diagnostics, but runtime admission would still not fail solely because they are absent.
3. `RequiredForAdmission`
   Future stricter mode where missing or invalid facts would block typed-slot admission before the runtime proceeds to the mainline legality path. At the bridge policy layer this is represented only as the reserved `CompilerTypedSlotPolicy.RequiredForAdmissionFuture` future seam, and it is not selectable in the current runtime mainline.

Only stage 1 is active today. The later stages are vocabulary for future tightening, not a claim that the repository already enforces them.

## Relationship to compiler preflight and runtime legality

Compiler-side preflight is intentionally stricter than the current runtime staging status:

- `HybridCpuBundleBuilder.Classify(...)` treats invalid emitted facts as a primary compiler inadmissibility result;
- `AdmissibilityClassification.StructurallyAdmissible` therefore means compiler preflight passed;
- this does not imply that the runtime already requires facts for correctness;
- `LegalityDecision` delivered through `IRuntimeLegalityService` remains the runtime authority surface.

This is an asymmetric contract by design. Compiler preflight can be fact-strict while runtime mainline still remains `ValidationOnly`.

## What this chapter is not claiming

This chapter does not claim that:

- typed-slot facts are already mandatory for canonical execution;
- missing facts currently fail Stage A admission;
- compiler hints outrank runtime legality;
- `WarnOnMissing` or `RequiredForAdmission` are already implemented as active runtime policy.

## Current limitations

- The staged vocabulary is now explicit, but only `ValidationOnly` is active.
- Runtime still accepts canonical execution without mandatory compiler-emitted typed-slot facts.
- No stricter missing-facts policy should be described as current behavior until code and proof suites change accordingly.

## Open questions / future work

- Whether `WarnOnMissing` should become an intermediate rollout mode before any fail-close transition.
- Which runtime entry points would need to fail closed if `RequiredForAdmission` ever becomes real.
- How to expose staged status in external compiler tooling without implying stronger runtime guarantees than the repository currently proves.
