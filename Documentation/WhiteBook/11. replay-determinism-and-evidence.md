# Replay, Determinism, and Evidence

## Goal of this chapter

This chapter describes the current replay model, the determinism envelope used by the runtime, and the evidence surfaces that report replay behavior instead of hiding it. The repository-level closure artifact is `HybridCPU_ISE/docs/replay-envelope.md`.

## Replay phase substrate

The replay substrate is centered on `LoopBuffer`:

- fixed-size cache for one 8-slot decoded bundle
- states: `Empty`, `Loading`, `Active`, `Draining`
- replay keyed by cached PC
- decode-once-replay-many behavior for loop-heavy and strip-mined execution

The loop buffer records not only replay hits but replay-phase structure:

- epoch count
- epoch length
- stable donor mask
- class donor capacities
- invalidation reason

## Determinism envelope

Current replay determinism is best described as an evidence-bounded envelope rather than as a blanket claim that every hidden state dimension is frozen.

Within that envelope, the runtime tries to preserve:

- stable donor structure
- replay-aware lane reuse
- replay certificate reuse
- class-template reuse
- deterministic transition accounting

Outside that envelope, the runtime records invalidation or mismatch rather than silently pretending the phase is identical. This is a replay-stable placement claim, not a global determinism theorem.

## Replay-aware scheduler behavior

The scheduler consumes replay context through `SetReplayPhaseContext(...)`. Replay affects:

- lane choice via `DeterministicLaneChooser.SelectWithReplayHint(...)`
- class-template reuse
- legality-template reuse
- assist invalidation on replay deactivation

Serializing boundaries and replay-phase changes invalidate reuse state deliberately.

## Replay anchors and bundle metadata

Replay behavior is also tied to bundle metadata. `BundleMetadata` contains `IsReplayAnchor`, and `ReplayAnchorEvaluator` uses this as part of replay-oriented diagnostics and snapshot-placement logic.

The important architectural point is that replay awareness is not bolted on after execution. Bundle metadata, scheduler state, and diagnostics all participate.

## Replay token

`ReplayToken` provides a more explicit deterministic reproduction and rollback carrier. It captures:

- vector configuration fields
- random seed
- trace hash
- pre-execution register state
- pre-execution memory state

Memory capture and restore are fail-closed:

- range materialization must be exact
- partial snapshotting/restoration is rejected
- rollback restores through explicit register and memory write paths

This is evidence that rollback and replay are treated as correctness-sensitive, not as best-effort convenience.

## Evidence surfaces

`ReplayEngine.Analysis.cs` provides evidence-oriented aggregation and comparison:

- `CompareReplayPhaseBehaviorWithinEnvelope(...)`
- `SummarizeReplayPhaseEvidence()`
- `SummarizeReplayEpochEvidence()`

These APIs confirm that replay is measured and compared explicitly. The architecture does not treat replay merely as a hidden performance optimization.

## Determinism philosophy

The current codebase supports a specific determinism philosophy:

- deterministic behavior is pursued where the runtime has explicit structure to preserve;
- deviations are surfaced as invalidations, misses, or evidence counters;
- replay-sensitive scheduling choices are allowed, but only through explicit policy and telemetry;
- runtime does not claim magic exactness when the relevant state is not actually preserved.

## Current limitations

- The current replay model is an envelope-based determinism model, not a claim that every conceivable hidden state is externally formalized.
- Replay token binding depends on explicit main-memory materialization and is fail-closed if that surface is absent.
- Replay evidence is strong in the ISE, but not yet packaged as a standalone external verification standard.

## Open questions / future work

- Keep `HybridCPU_ISE/docs/replay-envelope.md` aligned with replay invalidation and guard-before-reuse proof surfaces.
- Stronger export of replay evidence to offline tools.
- More explicit public specification of which state dimensions are guaranteed replay-stable.
