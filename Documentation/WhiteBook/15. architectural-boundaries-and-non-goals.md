# Architectural Boundaries and Non Goals

## Goal of this chapter

This chapter states clearly what the current HybridCPU ISE architecture is not, so that runtime scaffolding, research hooks, and historical aspirations are not misread as implemented production behavior.

## Non goals of the current mainline

The current architecture is not:

- a hidden out-of-order superscalar scheduler disguised behind a historical `FSP` label
- a nondeterministic lane-placement machine
- an ISA where typed-slot policy is encoded as a visible binary contract in slot bits
- a design where stale compiler metadata is silently accepted as correctness authority
- a complete hardware-rooted attestation platform
- a DBT-based compatibility front-end
- a machine where every optional historical opcode contour is still accepted in canonical decode
- a machine where opcode allocation, parser acceptance, metadata, or compiler facts imply execution support
- a machine where bounded `FENCE` / `FENCE_I` support implies cache/TLB/DMA/global-coherence support
- a VMX backend authority model
- a VMCS field-store model
- a system where DSC2, queue lifecycle, async overlap, coherent DMA/cache, or broad compiler lowering opens without an explicit gate
- a universal L7-SDC or external accelerator device protocol
- a SecureCompute backend runtime success path
- a VMX-owned SecureCompute model
- a CHERI, tagged-memory, or capability-aware `LOAD`/`STORE`/`FETCH` reinterpretation

These are not merely stylistic statements. Current code actively rejects several of these directions.

## Frontend boundary

The active frontend is native VLIW only. Compatibility ingress remains quarantined to compat or diagnostic seams, and DBT or scalar-generalized decode are not active proof surfaces in the current repository.

## SMT packing boundary

Historical `FSP` terminology should not be used as the architecture-defining label. In the live runtime, the mainline packing path is:

- class-capacity aware
- legality-governed
- replay-aware
- deterministic in lane materialization

So the architecture is not "best effort speculation into random empty slots." The canonical framing is bundle-compositional SMT packing under typed-lane legality.

## Compiler boundary

The architecture is not a compiler-only exact micro-packer. Current runtime owns:

- legality
- dynamic rejection
- lane choice
- replay-aware reuse
- assist quota and backpressure

Compiler structure matters, but runtime remains authoritative.

## Decode boundary

Canonical decode is not a permissive compatibility shell. `VliwDecoderV4` rejects:

- prohibited opcodes
- unknown opcodes
- retained branch compat contours
- retained move compat contours
- retired policy-gap bit usage
- unsupported residual scalar contours and future MatrixTile extensions beyond
  the closed four-op ABI
- unsupported `FENCE` / `FENCE_I` masks, payloads, flags, and sideband forms
- cache/TLB/coherency rows that remain reserved

So the current architecture is not "decode whatever legacy payload appears and guess the intent." Compat ingress is a retained boundary seam, not a second active frontend.

## ISA-surface boundary

The current instruction surface is closed for named contours and fail-closed elsewhere. Closed does not mean "all related opcodes are now production features." In particular:

- scalar repair is closed through `ZEXT.W`, while optional scalar helpers remain future scope;
- LR/SC and AMO W/D are closed through retire semantics and acquire/release ordering, while compiler atomic lowering remains closed;
- `FENCE` and `FENCE_I` are closed only for canonical zero-payload bounded runtime semantics;
- `SFENCE.VMA`, `DCACHE_CLEAN`, `DCACHE_INVAL`, `DCACHE_FLUSH`, and `ICACHE_INVAL` remain reserved;
- `VGATHER` / `VSCATTER` remain descriptor-only;
- matrix rows remain optional-disabled / decoder-rejected;
- lane6/lane7 coverage beyond current gated contours remains future-gated.

## Safety boundary

The current safety and legality stack is not yet a full production-proof external trust story. There is proof generation and signing scaffolding, but the signing path is simulated in the ISE.

Therefore the WhiteBook should not describe current proof objects as if they were already backed by secure hardware root-of-trust deployment.

## Execution-surface boundary

The current runtime is also not a claim that every potential execution surface is always wired:

- unsupported stream-control surfaces are guarded
- VMX routing is guarded when the execution plane is not wired
- optional contours may exist in the codebase as rejected surfaces rather than as mainline execution paths
- descriptor-only and carrier-only instruction rows can be valid transports without being executable ISA

## Hard specialized non-goals

Virtualization non-goals:

- no `VmxExecutionUnit`, `VmcsManager`, `IVmcsManager`, active VMCS pointer state, mutable VMCS field store, or legacy VMX runtime authority;
- no VMREAD interpretation as a backend VMCS state read;
- no successful VMCALL backend publication without a neutral hypercall backend owner;
- no VMX/VMCS/`VmxCaps` SecureCompute activation, grant, checkpoint, migration, materialization, or ownership.

Stream/lane6/lane7 non-goals:

- no DSC2 execution, queue lifecycle, async overlap, coherent DMA/cache, partial-success mode, VMX fallback, or broad production lowering from parser/model/helper evidence alone;
- no interpretation of MTILE load/store as ordinary LSU or DSC work;
- no StreamEngine completion, SRF valid/dirty state, execute capture,
  telemetry, or compiler metadata as MatrixTile architectural authority;
- no L7-SDC fallback into lane6, StreamEngine, VectorALU, assist helpers, `CustomAcceleratorMicroOp`, registry fixtures, or MatMul fixtures;
- no use of telemetry, replay evidence, certificates, tokens, helper observations, or backend diagnostics as authority.

SecureCompute non-goals:

- no completed secure backend runtime story;
- no secure VMCS, VMX mode, `VmxCaps` authority, CHERI ISA, tagged memory, capability registers, or capability-aware ordinary memory execution;
- no Stage A rejection of ordinary execution just because a SecureCompute descriptor is absent, disabled, or normalized to disabled.

## Scope boundary

HybridCPU ISE is an emulator/runtime codebase. This WhiteBook describes the current emulator/runtime architecture. It is not a final silicon product brief, not a compiler marketing document, and not a guarantee that all historical roadmap text remains current.

## Current limitations

- Some of these non-goal boundaries are enforced by rejection rather than by removal of all historical scaffolding.
- The codebase still contains compatibility and research seams that require careful reading.

## Open questions / future work

- Which optional or rejected contours should eventually become supported mainline behavior.
- Which scaffolding should be deleted rather than kept as guarded historical context.
