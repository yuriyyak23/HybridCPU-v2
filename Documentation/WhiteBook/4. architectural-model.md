# Architectural Model

## Goal of this chapter

This chapter fixes the current top-level architectural model of HybridCPU ISE: scalar control plus fixed-width VLIW issue, streaming-vector transport, and 4-way SMT inside a core.

## Scalar + VLIW + streaming vector

The current runtime is neither a plain scalar emulator nor a classic vector-register machine. It is best modeled as a compound architecture with three cooperating layers:

1. Scalar control and state management
2. Fixed 8-slot VLIW bundle transport
3. Streaming/vector execution with memory-to-memory semantics and RVV-influenced policy bits

The scalar layer remains responsible for conventional control, system, CSR, and retire-visible scalar work. The VLIW layer exposes explicit bundle structure. The vector side uses stream-oriented transport fields such as `VectorDataLength`, `Stride`, `StreamLength`, and `RowStride`, instead of presenting itself as a conventional wide SIMD register-file ISA only.

The instruction-refactor closure makes the scalar side more precise than the older high-level wording: the mandatory scalar repair surface is closed through `ZEXT.W = 321`, and the live compiler tail for that tracked scalar surface is also closed. Optional scalar helpers outside that contour remain future scope.

## Fixed 8-lane topology

The current typed-slot lane topology is fixed in code:

- lanes 0..3: `AluClass`
- lanes 4..5: `LsuClass`
- lane 6: `DmaStreamClass` or `MatrixTileStreamClass`
- lane 7: `BranchControl` or `SystemSingleton`

Lane 7 is explicitly aliased between branch/control and system-singleton work. That alias is part of the architectural scheduling model and not just an implementation accident.
Lane 6 is likewise physically aliased between DSC/assist transport and
MatrixTile memory transport. Its combined capacity is one operation.

Stream and external-device contours sit on this topology rather than redefining it:

- lane6 admits `DmaStreamClass` for DSC/assist transport or
  `MatrixTileStreamClass` for MTILE load/store.
- lane7 is the branch/system alias; L7-SDC uses the `SystemSingleton` side and must not borrow branch-control authority.
- StreamEngine, SRF, BurstIO, VectorALU, and assists are distinct helper or
  transport surfaces. Physical sharing does not merge DSC, MatrixTile, assist,
  or L7 descriptors and publication authority.

## 4-way SMT inside a core

Each physical core exposes `SmtWays = 4`. Intra-core SMT is not modeled as a separate queueing subsystem outside the bundle model; instead, sibling virtual threads can be co-packed into one bundle when legality permits.

Important nuance:

- SMT is real in the runtime model.
- The `VLIW_Instruction.VirtualThreadId` field is not the execution authority.
- Execution ownership comes from runtime state, nomination state, owner context, and legality checks.

## Typed-slot worldview

The typed-slot model is the mainline internal execution worldview. Each operation is classified by required slot class and pinning kind. The compiler may reason about classes, but the runtime remains authoritative for live admissibility.

This produces a system with:

- structural capacity at class level;
- deterministic lane materialization at runtime;
- explicit reject taxonomy instead of opaque failure;
- replay-aware lane reuse and class-template reuse.

## Current ISA-surface closure

The current runtime should be read as a closed implementation for specific published contours, not as an implicit promise that every historical opcode row is live. The closed runtime ISA surface currently includes:

- scalar integer64 repair through `ZEXT.W`;
- branch/control target transport through canonical `Immediate`;
- typed scalar load/store signedness, truncation, and retire-owned store commit;
- LR/SC and AMO W/D retire semantics;
- atomic acquire/release ordering for the currently published carriers;
- canonical zero-payload `FENCE` and `FENCE_I` as bounded lane7/system retire events;
- scoped audited 1D vector contours.

Adjacent surfaces remain intentionally non-executable or future-gated:
executable `VGATHER` / `VSCATTER`, lane6/lane7 coverage beyond current gated
contours, cache/TLB/coherency instructions, and broad compiler emission for
those contours. MatrixTile itself is closed runtime ISA, not future scope.

The current code also contains narrower stream/lane7 contours than older high-level wording suggested. Lane6 `DmaStreamCompute` has a typed descriptor carrier, parser, token store, helper/runtime execution-to-commit-pending path, and a current Phase 06 DSC1 materialized micro-op contour in `CloseToRTL`. That does not open DSC2, queue lifecycle, async overlap, coherent DMA/cache, partial-success completion, VMX guest lane binding, or broad compiler lowering. L7-SDC has lane7 system-device command carriers and runtime/model token/commit machinery; it remains separate from legacy custom accelerators and does not imply a universal external accelerator protocol.

MatrixTile is a separate execution plane:

- `MTILE_LOAD` and `MTILE_STORE` retain memory-semantic ISA metadata but own
  `MatrixTileMemory` resources and lane6 `MatrixTileStreamClass` placement;
- `MTILE_MACC` and `MTRANSPOSE` own `MatrixTileCompute` resources;
- MatrixRegisterFile state is architectural;
- StreamEngine/SRF transfer and execute capture are not architectural;
- retire is the only tile, accumulator, transpose, or store publication owner.

## Neutral authority overlays

Virtualization and SecureCompute are modeled as neutral runtime authority overlays:

- Virtualization authority belongs to runtime domain, memory, I/O, capability, evidence, trap, completion, nested, and retire owners. VMX can name frozen ABI/projection results and denied aliases after neutral admission; it cannot own the authority.
- VMCS/VMCSv2 is a projection vocabulary. Current VMREAD projection is partial and generated/read-only through neutral owners, while writes and ownerless privileged/control/host fields are denied.
- SecureCompute belongs to `SecureComputeDomainDescriptor`, `DomainRuntimeContext`, secure policies, and Layer 2 grant discipline. VMX, VMCS, `VmxCaps`, VMREAD, VMWRITE, CHERI-style registers, and tagged memory are not SecureCompute owners.

## What the model is not

The current architectural model is not:

- an out-of-order speculative superscalar machine;
- a free-form "any op in any lane" topology;
- a design where compiler hints silently override runtime legality;
- a design where SMT co-issue is invisible or nondeterministic.

The runtime is wide and cooperative, but explainability and structural legality remain first-class.

## Ownership and domain context

The current runtime distinguishes several identities:

- virtual thread identity;
- owner thread identity;
- owner context identity;
- domain tag;
- optional core/pod identity for inter-core assist transport.

This matters because legality and isolation are not determined by slot payload alone. The model is execution-context aware, not just opcode aware.

## Current limitations

- The lane topology is fixed; there is no evidence in current code of a dynamic or pluggable lane map.
- The runtime contains vector and stream transport, but not every possible optional contour is live in canonical decode and execute surfaces.
- Canonical `FENCE` / `FENCE_I` do not imply cache, TLB, DMA, or global coherence support.
- The architecture is described here from the ISE runtime viewpoint, not as a final hardware publication spec.

## Open questions / future work

- Whether future revisions introduce more slot classes or a wider topology.
- Whether future MatrixTile opcodes extend the closed four-instruction contour.
