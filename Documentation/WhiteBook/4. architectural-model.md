# Architectural Model

## Goal of this chapter

This chapter fixes the current top-level architectural model of HybridCPU ISE: scalar control plus fixed-width VLIW issue, streaming-vector transport, and 4-way SMT inside a core.

## Scalar + VLIW + streaming vector

The current runtime is neither a plain scalar emulator nor a classic vector-register machine. It is best modeled as a compound architecture with three cooperating layers:

1. Scalar control and state management
2. Fixed 8-slot VLIW bundle transport
3. Streaming/vector execution with memory-to-memory semantics and RVV-influenced policy bits

The scalar layer remains responsible for conventional control, system, CSR, and retire-visible scalar work. The VLIW layer exposes explicit bundle structure. The vector side uses stream-oriented transport fields such as `VectorDataLength`, `Stride`, `StreamLength`, and `RowStride`, instead of presenting itself as a conventional wide SIMD register-file ISA only.

## Fixed 8-lane topology

The current typed-slot lane topology is fixed in code:

- lanes 0..3: `AluClass`
- lanes 4..5: `LsuClass`
- lane 6: `DmaStreamClass`
- lane 7: `BranchControl` or `SystemSingleton`

Lane 7 is explicitly aliased between branch/control and system-singleton work. That alias is part of the architectural scheduling model and not just an implementation accident.

## 4-way SMT inside a core

Each physical core exposes `SmtWays = 4`. Intra-core SMT is not modeled as a separate queueing subsystem outside the bundle model; instead, sibling virtual threads can be co-packed into one bundle when legality permits.

Important nuance:

- SMT is real in the runtime model.
- The `VLIW_Instruction.VirtualThreadId` field is not the execution authority.
- Execution ownership comes from runtime state, nomination state, owner context, and legality checks.

## Typed-slot worldview

The typed-slot model is the mainline internal execution worldview. Each operation is classified by required slot class and pinning kind. The compiler may reason about classes, but the runtime remains authoritative for live admissibility.

This produces a system with:

- structural capacity at class level;
- deterministic lane materialization at runtime;
- explicit reject taxonomy instead of opaque failure;
- replay-aware lane reuse and class-template reuse.

## What the model is not

The current architectural model is not:

- an out-of-order speculative superscalar machine;
- a free-form "any op in any lane" topology;
- a design where compiler hints silently override runtime legality;
- a design where SMT co-issue is invisible or nondeterministic.

The runtime is wide and cooperative, but explainability and structural legality remain first-class.

## Ownership and domain context

The current runtime distinguishes several identities:

- virtual thread identity;
- owner thread identity;
- owner context identity;
- domain tag;
- optional core/pod identity for inter-core assist transport.

This matters because legality and isolation are not determined by slot payload alone. The model is execution-context aware, not just opcode aware.

## Current limitations

- The lane topology is fixed; there is no evidence in current code of a dynamic or pluggable lane map.
- The runtime contains vector and stream transport, but not every possible optional contour is live in canonical decode and execute surfaces.
- The architecture is described here from the ISE runtime viewpoint, not as a final hardware publication spec.

## Open questions / future work

- Whether future revisions introduce more slot classes or a wider topology.
- Whether matrix or other optional execution surfaces graduate from rejected contours into mainline architectural carriers.
