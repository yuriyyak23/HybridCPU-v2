# Open Research Directions

## Goal of this chapter

This chapter separates open or research-grade directions from current mainline implementation so they are not confused with already-landed architecture.

## Research versus implemented mainline

Mainline implementation in the current repository includes typed-slot admission, legality authority, replay-aware densification, and telemetry export. Research directions begin where code exists only as scaffolding, optional instrumentation, or obviously incomplete extension points.

## Shadow-oracle scheduling analysis

The scheduler contains `EnableShadowOracle` support and records oracle-gap snapshots. This is evidence of an exploratory validation/tuning direction, not of a production authority path.

Current reading:

- mainline scheduler decisions remain authoritative without the oracle;
- oracle machinery exists to compare or diagnose what the scheduler did versus a stronger hypothetical chooser.

## Counterexample-guided refinement

The code and README together suggest interest in refining masks, templates, and structural rules based on evidence. This fits naturally with:

- replay evidence
- legality reject classification
- certificate pressure metrics

But the current repository does not present a fully automated counterexample-guided refinement loop as mainline functionality.

## Stronger compiler-side typed-slot adoption

The runtime already validates typed-slot facts, but the compiler side is still not the only correctness path. A clear open direction is moving from:

- runtime-ready typed-slot agreement

to

- compiler-emitted typed-slot facts as standard output
- stronger compiler-side bundle reasoning at class level

This is one of the most concrete next-step research-to-product transitions in the codebase.

## Stronger attestation and proof packaging

The legality and proof surfaces already exist, but stronger work remains open:

- hardware-rooted or externally verifiable proof signing
- clearer proof schemas
- better publication of replay and legality evidence

At present this is readiness scaffolding, not a finished deployment story.

## Richer assist-plane cooperation

The assist runtime already models donor taxonomy in a fine-grained way. Open directions include:

- automatic intra-core cross-VT donor generation
- richer adaptive lane-6 versus LSU carrier policy
- stronger inter-core assist coordination policy

These are plausible future extensions because the taxonomy and legality matrix already exist, but they are not yet the default mainline behavior.

## Test and evidence infrastructure

Validation is substantial, and the repository now declares a VSTest-based runner path that supports test listing and targeted execution through the standard `dotnet test` entrypoint. A practical open direction is making broader evidence generation and smoke/full-suite validation baselines easier to reproduce automatically.

## What is authoritative in code

Primary authority sources:

- guarded or optional paths in `HybridCPU_ISE/Core/Pipeline/Scheduling/`
- proof-oriented surfaces in `HybridCPU_ISE/Core/Pipeline/Safety/`
- assist taxonomy in `HybridCPU_ISE/Core/Pipeline/Assist/`
- contract comments in `HybridCPU_ISE/Core/Contracts/CompilerContract.cs`

## Current limitations

- Not every research direction in this chapter has a formal design document in the current tree.
- Some older roadmap references are missing from the checkout, so research framing must be inferred carefully from live code and guarded features.

## Open questions / future work

- Which exploratory scheduler features will graduate into supported mainline behavior.
- Whether the compiler contract, proof system, and evidence model can be published as standalone specs.
- How aggressively historical compatibility scaffolding should be pruned as the architecture matures.
