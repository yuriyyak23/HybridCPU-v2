# Legality Predicate

## Goal of this chapter

This chapter externalizes the canonical legality decomposition for the active typed-slot SMT path. Its job is not to invent a new mechanism, but to state one repository-facing contract that already matches the live scheduler, legality service, replay substrate, and proof suites.

## What object represents legality

The canonical legality object is `LegalityDecision`.

The active scheduler does not treat raw masks, ad hoc certificate peeks, or lane masks as the top-level truth surface. It consumes `LegalityDecision` values through `IRuntimeLegalityService`.

For the active typed-slot SMT path, the main authority sources are:

- `GuardPlane`
- `ReplayPhaseCertificate`
- `StructuralCertificate`

`DetailedCompatibilityCheck` and `AdmissionMetadataStructuralCheck` remain compatibility or auxiliary authority sources. They should not be presented as the mainline typed-slot SMT legality contract.

## Canonical Stage A / Stage B split

The active scheduler boundary is:

- Stage A: `TryClassAdmission(...)`
- Stage B: `TryMaterializeLane(...)`

No primary repository-facing surface should collapse these into one opaque "admission" step.

Stage A decides whether a candidate may advance under the current legality and dynamic-state envelope. Stage B only materializes one concrete lane from the already admitted class/topology envelope.

## Canonical legality decomposition

For the active typed-slot SMT path, the legality predicate is truthfully decomposed as follows.

### Stage A: class admission

`TryClassAdmission(...)` checks, in current code order:

1. class-capacity availability through `SlotClassCapacityState`
2. runtime legality through `IRuntimeLegalityService.EvaluateSmtLegality(...)`
3. outer-cap dynamic gates through `TryPassOuterCap(...)`

There is no extra pre-stage before Stage A beyond preparing the current replay/template context with `PrepareSmt(...)`. Guard plane checks are the first checker-owned substep inside the Stage A legality-service decision, not a separate lane-materialization or post-admission patch.

### Stage A legality-service substeps

The runtime legality substage is checker-owned and returns one `LegalityDecision`.

Its active SMT order is:

1. `EvaluateSmtBoundaryGuard(...)`
2. owner-context and domain guard checks from bundle metadata
3. replay-phase certificate reuse if the live template matches
4. current-bundle structural witness fallback otherwise

This means guard rejection precedes replay reuse, and replay reuse precedes
structural witness fallback.

## Certificate and replay semantics

`BundleResourceCertificate4Way` is the active SMT certificate substrate. It carries:

- shared non-register resource state
- per-VT register-group hazard state
- class occupancy state

`BundleResourceCertificate4Way.StructuralIdentity` is the canonical replay/template compatibility seam. The scheduler should be read as comparing certificate identity through the legality service and replay substrate, not as manually interpreting raw mask layout everywhere.

For the active SMT path, replay/template identity is keyed by:

- `ReplayPhaseKey`
- `StructuralIdentity`
- `SmtBundleMetadata4Way`
- `BoundaryGuardState`

This key is carried through `PhaseCertificateTemplateKey4Way` and coordinated by `PrepareSmt(...)`, `EvaluateSmtLegality(...)`, `RefreshSmtAfterMutation(...)`, and `InvalidatePhaseMismatch(...)`.

The companion artifact `23. certificate-semantics-and-legality-evidence.md` gives the paper-facing `BundleLegalityDescriptor` section and the single legality-evidence diagram linking decoded bundle facts, certificate materialization, structural identity, runtime legality decision, lane selection, and replay reuse.
The wider pipeline state tuple and cycle-step relation now live in `Documentation/operational-semantics.md`.

## Stage B: lane materialization

`TryMaterializeLane(...)` runs only after Stage A has returned success.

Its live constraints are:

- hard-pinned candidates must find `PinnedLaneId` free
- class-flexible candidates use `SlotClassLaneMap.GetLaneMask(...)`
- eligible lanes are intersected with current free lanes
- replay-aware stable-donor masking may narrow the eligible set further
- `DeterministicLaneChooser.SelectWithReplayHint(...)` selects the final lane

If no legal lane remains, Stage B returns `PinnedLaneConflict` or `LateBindingConflict`. This is a lane-materialization failure, not a reopening of the Stage A legality decision.

## Current status of typed-slot facts

Compiler-emitted typed-slot facts remain a validated transport and agreement surface, not a mandatory correctness substrate for canonical execution. `ValidateTypedSlotFacts(...)` and `VerifyTypedSlotAgreement(...)` are therefore part of the legality story, but they do not currently replace runtime legality authority.
The dedicated artifact `22. typed-slot-contract-staging.md` freezes this as the current `ValidationOnly` mode and reserves stronger future stages without claiming they already landed.
