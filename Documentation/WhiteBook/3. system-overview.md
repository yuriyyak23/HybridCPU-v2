# System Overview

## Goal of this chapter

This chapter explains how the current repository is partitioned into runtime, compiler, test, and documentation surfaces, and how the main runtime subsystems fit together.

## Repository-level picture

The current top-level workspace contains several distinct subsystems:

- `HybridCPU_ISE/`: emulator/runtime and architectural implementation
- `HybridCPU_Compiler/`: compiler-side code and transport artifacts
- `HybridCPU_ISE.Tests/`: test suites covering decode, execution, scheduling, replay, telemetry, and safety
- `CpuInterfaceBridge/`: bridge surfaces used by the test and tooling stack
- `Documentation/`: documentation workspace, including the WhiteBook

The WhiteBook is primarily about the first and third entries, with compiler contract references where the runtime exposes them explicitly.

## Runtime system shape

Within `HybridCPU_ISE/`, the live architecture is distributed across a few critical areas:

- `Arch/`: native VLIW slot and bundle carriers, plus quarantined compat ingress and serialization seams
- `Core/Decoder/`: native-VLIW-only active front-end and canonical IR publication
- `Core/Legality/`: bundle-local legality extraction and typed-slot facts
- `Core/Pipeline/`: scheduler, safety verifier, replay, assist runtime, and pipeline metadata
- `Core/Diagnostics/`: trace, telemetry, replay evidence, and exporters
- `Core/Memory/`: memory-side runtime contracts and scoreboarding
- `Processor/`: processor integration and cycle-by-cycle execution shell

## End-to-end runtime flow

The live path from bundle bytes to visible retirement is:

1. Fetch native 256-byte VLIW bundle or replay cached decoded slots.
2. Decode raw slots into `InstructionIR` and `DecodedInstructionBundle`.
3. Build legality and typed-slot descriptors.
4. Optionally derive a foreground issue plan using FSP and SMT densification.
5. Materialize issue-packet lanes into EX/MEM/WB pipeline state.
6. Execute, access memory, retire results, and publish trace/telemetry/evidence.

Important architectural point: the runtime preserves a distinction between canonical decode state and derived issue layout. Densification is modeled as a runtime-derived issue plan, not as a mutation of the underlying architectural bundle description.

The active frontend is native VLIW only. Compatibility ingress remains quarantined to compat or diagnostic seams, and DBT or scalar-generalized decode are not active proof surfaces in the current repository.

## Sideband metadata model

The system uses an explicit split between:

- architectural VLIW payload;
- non-architectural slot and bundle metadata;
- runtime-only legality and evidence objects.

Examples:

- `VliwBundleAnnotations` carries per-slot `InstructionSlotMetadata`.
- `BundleMetadata` carries `FspBoundary`, `IsReplayAnchor`, and diagnostic tags.
- `BundleResourceCertificate4Way` carries runtime structural legality state.

This split is one of the main modernization themes in the codebase: correctness is no longer meant to depend on legacy policy encodings in raw slot bits.

## Core architectural services

The runtime repeatedly reuses a few key authority services:

- `VliwDecoderV4`: fail-closed canonical decode
- `BundleLegalityAnalyzer`: structural bundle classification
- `MicroOpScheduler`: typed-slot admission, densification, fairness, and assist injection
- `SafetyVerifier` and `IRuntimeLegalityService`: legality decisions and guard checks
- `LoopBuffer`: replay-phase substrate for decode-once-replay-many
- `TelemetryExporter` and trace infrastructure: evidence and measurement export

## Compiler relationship

The runtime explicitly references a versioned compiler contract through `CompilerContract.Version`. However, the current runtime does not require non-default slot metadata for correct execution of canonical opcodes. That is a deliberate boundary: compiler cooperation is valuable, but runtime correctness remains authoritative.

- Historical notes still exist in the repository, but current authority is intentionally concentrated in the README, WhiteBook, and named validation artifacts.
- The system overview here is architecture-focused; it does not attempt to document every UI, tooling, or experimental directory.
