# VLIW Binary Format

## Goal of this chapter

This chapter defines the authoritative slot and bundle container format used at the HybridCPU ISE front door, including bit layout, bundle shape, streaming fields, SMT transport hints, and fail-closed compatibility boundaries.

## Slot container: `VLIW_Instruction`

The base instruction carrier is `HybridCPU_ISE.Arch.VLIW_Instruction`, declared as a fixed-width packed struct with `StructLayout(LayoutKind.Sequential, Pack = 1)`. Each slot occupies exactly 32 bytes, implemented as four 64-bit words:

- `word0`
- `word1`
- `word2`
- `word3`

The format is container-centric rather than category-centric. The opcode in `word0[63:48]` is an opaque 16-bit field. Architectural classification does not come from bit slicing the opcode; it comes from `OpcodeRegistry` and `InstructionClassifier`.

## Bundle container: `VLIW_Bundle`

`HybridCPU_ISE.Arch.VLIW_Bundle` is an 8-slot aggregate:

- `inst0` through `inst7`
- 8 slots x 32 bytes = 256 bytes total

`TryWriteBytes` and `TryReadBytes` serialize and deserialize the full bundle as eight consecutive 32-byte slot images.

## Word layout

### `word0`

`word0` is the execution-control word:

- `[63:48]` `OpCode` (16 bits)
- `[47:40]` `Reserved` (8 bits)
- `[39:32]` `DataType` (8 bits)
- `[31:24]` `PredicateMask` (8 bits)
- `[23:16]` flags (8 bits)
- `[15:0]` `Immediate` (16 bits)

The flags in `word0[23:16]` are currently:

- bit 23: `Reduction`
- bit 22: `Is2D`
- bit 21: `Indexed`
- bit 20: `TailAgnostic`
- bit 19: `MaskAgnostic`
- bit 18: `Acquire`
- bit 17: `Release`
- bit 16: `Saturating`

This matters architecturally because these flags are the active correctness-bearing policy bits. Scheduling policy is not encoded here or anywhere else in the architectural slot payload.

### `word1`

`word1` is the primary source/addressing transport:

- full 64-bit view: `DestSrc1Pointer`
- `[31:0]` `SourceAPointer`
- `[63:32]` `SourceBPointer`

The naming reflects transport reuse rather than a single semantic role. Depending on opcode form, the same field may carry register-packed information or pointer-like transport values.

### `word2`

`word2` carries destination and vector-length transport:

- full 64-bit view: `Src2Pointer`
- `[31:0]` `DestinationPointer`
- `[63:32]` `VectorDataLength`

For canonical load/store absolute addressing in `VliwDecoderV4`, `Src2Pointer` is used first; the decoder falls back to sign-extended `Immediate` when `Src2Pointer == 0`.

### `word3`

`word3` is the stream and transport-hint word:

- `[63:51]` `RowStride` (13 bits)
- `[50]` retired legacy policy gap bit
- `[49:48]` `VirtualThreadId` (2 bits)
- `[47:16]` `StreamLength` (32 bits)
- `[15:0]` `Stride` (16 bits)

`RowStride` is range-checked to `<= 8191`. `StreamLength` and `Stride` remain direct transport values.

## Streaming semantics

The binary format includes first-class streaming transport fields:

- `StreamLength`
- `Stride`
- `RowStride`
- `VectorDataLength`
- `Indexed`
- `Is2D`

These fields confirm that streaming/vector behavior is part of the slot container itself. However, the current runtime still separates "container contains stream parameters" from "configured execution surfaces are wired for every possible stream-control contour". In particular, `CPU_Core.PipelineExecution.SurfaceContract.cs` explicitly rejects unsupported mainline stream-control execution surfaces when they reach live routing.

## SMT semantics in the container

`word3[49:48]` stores `VirtualThreadId`, but current code is explicit that this is a transport hint only:

- it is preserved for compat ingress/egress and diagnostics;
- it does not bind execution context;
- it must not influence functional decode.

The actual execution context is determined by runtime ownership and scheduler state, not by the slot payload hint.

## Fail-closed compatibility boundary: `word3[50]`

`word3[50]` is a retired legacy scheduling-policy bit. Current code treats it as a compatibility hazard, not as a live architectural flag.

The fail-closed boundary appears in three places:

1. `VLIW_Instruction.ValidateWord3ForProductionIngress(...)`
2. `VLIW_Instruction.ValidateRetiredPolicyGapBitForProductionIngress()`
3. `VliwDecoderV4.RejectLegacyPolicyGap(...)`

If this bit survives into a production stream, decode throws `InvalidOpcodeException`. `VLIW_Bundle.SetInstruction(...)` also validates the bit on ingress.

This is an important current architectural boundary:

- old scheduling-policy encodings are not silently tolerated;
- correctness no longer depends on a hidden policy bit in the binary payload;
- the runtime rejects stale carriers rather than attempting compat interpretation in the mainline decoder.

## NOP and empty-slot semantics

At bundle decode, `VliwDecoderV4.DecodeInstructionBundle(...)` treats `OpCode == 0` as the hardware NOP sentinel and emits an empty decoded slot. This is the current canonical empty-slot convention in the front-end path.

## Binary/metadata split

The architectural payload lives in `VLIW_Instruction` and `VLIW_Bundle`. Scheduling annotations, slot metadata, replay anchors, and fence-like bundle policy are carried separately through:

- `InstructionSlotMetadata`
- `VliwBundleAnnotations`
- `BundleMetadata`

That split is central to the modernized contract: execution correctness must not depend on retired policy bits in the raw slot payload.

## Current limitations

- The current WhiteBook does not treat `Reserved` bits as architecturally assigned.
- The container carries stream-related fields, but not every possible stream-control contour is wired into a live execution surface.
- `VirtualThreadId` remains a hint only; there is no ISA-visible guarantee that this field binds final runtime lane or thread ownership.

## Open questions / future work

- Whether reserved bits should ever be repurposed into externally stable metadata.
- Whether a stricter published binary-format spec should be extracted from the current code and frozen independently from the emulator source tree.
