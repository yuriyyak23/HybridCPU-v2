# SMT VLIW Packing and Densification

## Goal of this chapter

This chapter explains how the current runtime densifies partially occupied bundles through 4-way intra-core SMT packing while preserving typed-slot legality and deterministic behavior.

Historical terminology note: code and test names still use `FSP` as a retained alias in some places. In this WhiteBook, the canonical architecture language is SMT packing / densification rather than `FSP` as the class-defining name of the machine.

## Heterogeneous 8-wide issue

The runtime does not treat the bundle as eight identical scalar slots. Densification happens over a heterogeneous topology:

- 4 ALU-capable lanes
- 2 LSU lanes
- 1 DMA/stream lane
- 1 aliased branch/system lane

So "8-wide issue" in current HybridCPU ISE means "8 heterogeneous physical lanes with class-aware constraints," not "8 interchangeable scalar issue positions."

## Main densification path: `PackBundleIntraCoreSmt`

The core intra-core packing routine is `MicroOpScheduler.PackBundleIntraCoreSmt(...)`.

At a high level it:

1. starts from an owner bundle;
2. computes current class occupancy;
3. builds a `BundleResourceCertificate4Way`;
4. builds bundle metadata and boundary-guard state;
5. creates nomination state for sibling virtual threads;
6. ranks candidates using fairness and optional bank-pressure tie-breaks;
7. injects candidates through typed-slot admission and lane materialization;
8. optionally injects assists afterward.

This is the mainline densification surface for intra-core SMT.

## Structural legality during packing

Packing is bounded by several runtime authorities:

- `BundleResourceCertificate4Way`
- legality service decisions
- owner/domain guards
- replay-phase reuse constraints
- memory-budget and speculation-budget gates
- class-capacity state

This means densification is not just "insert any ready op into any hole." It is
legality-witness-governed and topology-aware.

## Deterministic ordering and fairness

The scheduler supports:

- credit-based fairness ranking
- legacy TDM-like ordering fallback
- bank-pressure tie-break for memory candidates
- deterministic lane choice after class admission

These mechanisms are runtime policy, but they are explicit and inspectable. The code does not implement a hidden free-form heuristic scheduler.

## Replay-aware densification

Replay context influences densification through:

- stable donor mask
- previous-lane reuse
- class-template reuse
- replay certificate reuse

In replay-active phases, candidate placement is filtered so the scheduler prefers reproducible donor structure and reproducible lane selection when still legal.

## Densification and explicit packet materialization

After packing, the derived bundle becomes an issue packet. Later pipeline code materializes that packet into concrete EX/MEM/WB lane state. This preserves an architectural separation between:

- "candidate was admitted into the bundle"
- "candidate became a concrete execution lane"

## Assist path is adjacent, not identical

Assists are injected after main SMT candidate injection and under a separate assist quota/backpressure model. This is important:

- densification is broader than assists;
- assist injection participates in typed-slot placement;
- but assist policy is not the same thing as core SMT candidate packing.

## Intra-core versus inter-core

Current code distinguishes:

- `PackBundleIntraCoreSmt(...)`: intra-core SMT packing
- `PackBundle(...)`: retained inter-core packing path (historically `FSP`-labelled)

The WhiteBook should not collapse them into one mechanism. The mainline "4-way SMT packing" chapter is about the first path.

## Current limitations

- Mainline densification is intra-core for SMT packing; the WhiteBook does not claim a general cross-core shared-bundle issue model.
- The scheduler still contains a legacy exact-slot path when `TypedSlotEnabled` is false.
- One aliased branch/system lane and class-specific capacities constrain achievable width.

## Open questions / future work

- Whether typed-slot mode becomes the only supported path.
- Whether the inter-core path should be documented as a separate densification model in future revisions.
- Whether fairness and pressure policy should be externalized as a more stable configuration contract.
