# Safety, Isolation, and Legality

## Goal of this chapter

This chapter describes the current fail-closed legality stack: certificates, guard planes, typed-slot agreement checks, domain isolation, and the authority model that decides whether cross-thread or assist injection is legal. The repository-level guard closure artifact is `HybridCPU_ISE/docs/domain-guard-invariants.md`.

## Legality as an explicit authority seam

The current scheduler does not treat raw certificates or mask intersections as the top-level source of truth. Instead, it consumes explicit `LegalityDecision` values produced by the legality layer.

Important current types:

- `LegalityDecision`
- `LegalityAuthoritySource`
- `RejectKind`
- `TypedSlotRejectClassification`

This is architecturally important because it makes legality provenance inspectable. The runtime can state whether a decision came from:

- `StructuralCertificate`
- `ReplayPhaseCertificate`
- `GuardPlane`
- `DetailedCompatibilityCheck`
- `AdmissionMetadataStructuralCheck`

The focused artifact `20. legality-predicate.md` is the repository-facing closure for this authority seam: it ties `LegalityDecision`, `IRuntimeLegalityService`, Stage A admission, guard-plane precedence, replay reuse, and Stage B lane materialization into one contract.
The companion artifact `23. certificate-semantics-and-legality-evidence.md` explains the paper-facing `BundleLegalityDescriptor`, certificate structural identity, and the single legality-evidence chain without making raw mask layout the central story.

## Structural certificate layer

`BundleResourceCertificate4Way` is the main legality certificate for intra-core SMT packing.

It separates:

- `SharedMask` for shared non-register resources
- `RegMaskVT0..RegMaskVT3` for per-VT register hazard groups
- `ClassOccupancy` for typed-slot occupancy

`CanInject(...)` checks:

1. shared resource conflict
2. per-thread register-group hazards

RAR is explicitly allowed; RAW, WAR, and WAW are the true register-group hazards.

## Structural identity and replay reuse

`BundleResourceCertificate4Way.StructuralIdentity` creates an opaque digest over:

- shared mask state
- per-VT register mask state
- class occupancy vector
- lane topology signature
- resource profile
- operation count

The architecture should be read as "certificate identity is the replay/reuse seam," not "the scheduler directly compares ad hoc mask fragments everywhere."
For the active SMT typed-slot path, replay/template matching is keyed by `ReplayPhaseKey`, `StructuralIdentity`, `SmtBundleMetadata4Way`, and `BoundaryGuardState`.
`23. certificate-semantics-and-legality-evidence.md` expands this into the descriptor-to-certificate-to-decision diagram used by Phase 02 closure.

## Guard plane

`SafetyVerifier.Guards.cs` adds guard-plane checks before certificate reuse:

- owner-context mismatch reject
- domain mismatch reject
- boundary guard reject
- inter-core domain guard reject
- ordering and non-stealable rejection in inter-core cases

The guard plane is therefore earlier than certificate reuse in the authority stack. In Phase 04 vocabulary this is the guard-before-reuse rule: domain and owner guards precede replay reuse and certificate acceptance.

## Domain isolation

The main domain-isolation switch is `KernelDomainIsolation`, which defaults to `true`.

Current behavior:

- when `DomainTag == 0`, kernel-to-user injection is denied if a user-domain certificate is active;
- otherwise domain compatibility is checked by intersection with the domain certificate;
- cross-domain mismatches surface as explicit guard rejects rather than falling through as generic failure.

This is a real architectural isolation boundary in current code, not just a diagnostic comment.

## Typed-slot agreement and validation

The legality layer also validates producer-side structural facts:

- `ValidateTypedSlotFacts(...)`
- `VerifyTypedSlotAgreement(...)`
- `ClassifyReject(...)`

These checks do not replace runtime authority, but they formalize the compiler/runtime handshake and separate static structural mismatch from dynamic runtime rejection.
The dedicated artifact `22. typed-slot-contract-staging.md` freezes the current staged status of those facts as `ValidationOnly`, rather than overstating them as already mandatory for canonical admission.

## Proof and verification surface

`SafetyVerifier` also includes a proof-oriented layer:

- `VerifyBundle(...)`
- `GenerateProof(...)`
- `ExportSMTLib2Proofs()`
- simulated proof signing through `SignProof(...)`

This is valuable as an evidence and readiness surface, but the WhiteBook should describe it accurately: current signing is simulated with SHA-256 over proof fields, not a production hardware root-of-trust implementation.

## Current limitations

- The proof-signing surface is simulated, not a production cryptographic attestation path.
- The legality system is runtime-local; it does not by itself constitute a complete external trust or TEE story.
- The WhiteBook does not claim that every proof-oriented helper is on the hot path of normal execution.

## Open questions / future work

- Stronger external attestation and hardware-rooted proof signing.
- Better separation between hot-path legality and offline verification tooling.
- Possible publication of the legality authority model as a standalone formal contract.
