# Repository Layout and Code Map

## Goal of this chapter

This chapter maps the current live repository layout to the architectural chapters in this WhiteBook and identifies the current authoritative entry points for architecture and validation reading.

## Top-level live layout

Current top-level directories include:

- `build/`
- `CpuInterfaceBridge/`
- `Documentation/`
- `evaluation_results/`
- `forms/`
- `HybridCPU_Compiler/`
- `HybridCPU_EnvGUI/`
- `HybridCPU_ISE/`
- `HybridCPU_ISE.Tests/`
- `TestAssemblerConsoleApps/`
- `TestResults/`

For the WhiteBook, the primary authority areas are `HybridCPU_ISE/`, `HybridCPU_ISE.Tests/`, and selected contract surfaces under `HybridCPU_Compiler/`.

## Runtime code map

Within `HybridCPU_ISE/`, the most relevant directories for architecture work are:

- `Arch/`: VLIW slot and bundle carriers, compat ingress, serialization
- `Core/Decoder/`: canonical decode and decoded-bundle contracts
- `Core/Legality/`: bundle legality analysis and typed-slot classification
- `Core/Pipeline/Scheduling/`: typed-slot admission, SMT packing, assists, fairness
- `Core/Pipeline/Safety/`: legality authority, domain guards, proof surfaces
- `Core/Pipeline/Certificates/`: structural certificate types and replay/template identity
- `Core/Pipeline/Components/`: loop buffer and replay substrate
- `Core/Pipeline/Metadata/`: bundle metadata and sideband policy transport
- `Core/Diagnostics/`: telemetry exporter, trace counters, replay evidence
- `Core/Memory/`: memory-side coordination and bank-visible behavior
- `Core/State/`: state constants such as SMT width
- `Processor/`: processor shell integration

## Compiler and bridge map

Current adjacent sources:

- `HybridCPU_Compiler/Core/` and `HybridCPU_Compiler/API/` for compiler-side artifacts
- `HybridCPU_ISE/Core/Contracts/` for explicit compiler/runtime transport and versioning
- `CpuInterfaceBridge/` for bridge-level integration

The WhiteBook does not treat every compiler directory as architecture authority, but it does use contract surfaces that the runtime explicitly enforces.

## Test map

`HybridCPU_ISE.Tests/tests/` is the primary validation surface. The file set includes:

- phase-numbered behavioral suites
- audit suites
- compatibility and fail-closed boundary suites
- telemetry and replay suites

This directory is the main source for the "confirmed by tests" parts of the WhiteBook.

## Documentation map and current entry points

The current documentation workspace contains:

- `README.md`
- `Documentation/WhiteBook/`
- `Documentation/ISE/`
- `Documentation/Research/`
- historical folders such as `Old/` and `old2/`

Current authoritative entry points are:

- `README.md` for the top-level repository reading order
- `Documentation/WhiteBook/` for the architecture-facing narrative
- `Documentation/validation-baseline.md` and `Documentation/evidence-matrix.md` for validation authority
- `Documentation/AsmAppTestResults.md` for the runtime sanity baseline

`README.md` is now a current authoritative entry point and links readers to the live architecture and validation documents.
Historical folders such as `Old/` and `old2/` remain non-authoritative repository archaeology.
